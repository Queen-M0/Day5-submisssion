# 言鉴 AI 前后端对接与后端开发契约

> 项目：AI 文字内容审核与申诉复核社区  
> 产品形态：单社区、多话题、线性楼层式文字讨论社区  
> 用途：说明当前前端 Demo 能力，并作为后端数据库、API、状态和权限开发的对接基准。  
> 更新日期：2026-07-16

---

## 1. 对接目标

当前前端已经依据 `product-prd/product-prd.md` 完成可交互 Demo，但多话题、楼层、申诉反证和人工复核状态暂时保存在前端本地数据中。

后端下一阶段不需要重新设计产品，而是按照本文契约提供数据库和 API，将前端本地状态替换成真实后端状态。

核心闭环：

```text
提交内容
→ 构建话题上下文
→ AI 初审
→ 证据真实性校验
→ 系统允许 / 限制 / 转人工
→ 用户查看原因并申诉
→ 申诉 AI 生成正反依据
→ 审核员最终裁决
→ 改判后恢复公开
→ 保存完整时间线
```

## 3. 当前后端已有能力与差距

### 3.1 当前已有表

```text
users
scenes
contents
moderation_records
appeals
manual_reviews
audit_logs
```

### 3.2 当前已有接口

```text
GET  /api/auth/demo-users
GET  /api/scenes
GET  /api/scenes/{sceneId}/contents
POST /api/contents
GET  /api/contents/{contentId}/moderation
POST /api/contents/{contentId}/appeals
GET  /api/me/appeals
GET  /api/reviewer/tasks
GET  /api/reviewer/tasks/{taskId}
POST /api/reviewer/tasks/{taskId}/decision
```

### 3.3 主要差距

1. 没有 `topics` 表，无法表达“固定社区 -> 多话题 -> 多楼层”；
2. `contents` 没有 `topic_id` 和 `floor_number`；
3. 申诉没有 `extra_context`；
4. 没有保存申诉反证 AI 的正反依据；
5. AI 建议动作和系统分流没有明确分开；
6. 审核结果缺少意图、对象、上下文标签和不确定点；
7. 没有“我的发布”接口；
8. 没有完整时间线接口；
9. 没有审核历史接口；
10. 当前人工决定枚举与新版前端不一致；
11. 当前 Seed 用户与前端演示身份不一致。

## 4. 推荐数据库设计

### 4.1 `scenes`：继续作为固定社区

MVP 只有一个固定社区，不要求将 `scenes` 重命名为 `communities`。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| id | varchar(36) PK | 固定社区 ID |
| type | varchar(30) | 建议使用 `community` |
| title | varchar(120) | 社区名称 |
| description | text | 社区介绍 |
| created_at | datetime | 创建时间 |

推荐 Seed：

```text
id: community-001
title: AI Native 青年讨论社区
type: community
```

### 4.2 `topics`：新增话题表

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| id | varchar(36) | PK | 话题 ID |
| scene_id | varchar(36) | FK, index | 所属固定社区 |
| author_id | varchar(36) | FK, index | 发起人 |
| title | varchar(120) | not null | 话题标题 |
| summary | varchar(255) | not null | 话题摘要 |
| category | varchar(40) | index | 话题分类 |
| status | varchar(30) | index | 话题状态 |
| visible_to_public | boolean | not null | 是否公开 |
| view_count | integer | default 0 | 浏览量 |
| created_at | datetime | not null | 创建时间 |
| updated_at | datetime | not null | 更新时间 |
| last_active_at | datetime | index | 最近公开楼层时间 |

业务规则：

- 发起话题时，同时创建 Topic 和 1 楼 Content；
- 1 楼审核通过后，话题才进入公开列表；
- 1 楼被限制或等待人工复核时，仅作者和审核员可见；
- 1 楼申诉成功后，话题恢复公开。

### 4.3 `contents`：增加话题和楼层字段

当前 `contents` 继续代表每一楼内容。

建议新增：

| 字段 | 类型 | 约束 | 说明 |
| --- | --- | --- | --- |
| topic_id | varchar(36) | FK, index | 所属话题 |
| floor_number | integer nullable | 与 topic 联合唯一 | 公开楼层号 |
| target_user_id | varchar(36) nullable | FK | 明确回复对象，可选 |

已有字段继续使用：

- `scene_id`：所属固定社区；
- `content_type`：建议使用 `topic_root` 或 `forum_reply`；
- `author_id`；
- `parent_id`：继续作为 `reply_to_content_id`；
- `text`、`normalized_text`；
- `status`、`visible_to_public`；
- `created_at`、`updated_at`。

建议索引：

```text
UNIQUE(topic_id, floor_number)
```

楼层号规则：

1. 审核中、限制中和人工复核中均不分配楼层号；
2. 审核通过时才执行 `MAX(floor_number) + 1`；
3. 申诉成功后按当前最新公开楼层追加；
4. 楼层号分配必须放在数据库事务中；
5. 公开页面不能因为受限内容出现楼层空号。

### 4.4 `moderation_records`：扩展 AI 输出

现有字段保留，建议新增：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| context_tags | JSON | quote、repeated_targeting 等标签 |
| intent | text | 表达意图 |
| target_user_ids | JSON | 涉及对象 ID |
| context_used | JSON | 实际使用的上下文 |
| uncertainties | JSON | AI 不确定点 |
| suggested_action | varchar(30) | AI 建议动作 |
| system_decision | varchar(30) | 系统实际分流 |
| model_version | varchar(50) | 模型版本 |
| rule_version | varchar(30) | 规则版本 |
| evidence_valid | boolean | 证据校验是否通过 |
| failure_reason | text nullable | AI 超时、非法 JSON 等原因 |

`evidence` 统一格式：

```json
[
  {
    "contentId": "content-005",
    "quote": "大家都看看他平时是什么样",
    "reason": "结合前文存在群体施压倾向",
    "verified": true
  }
]
```

必须分别保存：

```text
AI suggested_action
系统 system_decision
人工 final_decision
```

### 4.5 `appeals`：增加补充上下文和反证结果

建议新增：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| extra_context | text | 用户补充上下文 |
| counter_analysis | JSON | 申诉反证 AI 输出 |
| analyzed_at | datetime nullable | 反证完成时间 |

`counter_analysis` 格式：

```json
{
  "supportsOriginalDecision": ["内容确实包含攻击性表达"],
  "supportsChange": [
    "攻击性片段使用引号标记",
    "作者后半句明确表达反对",
    "用户补充了引用来源"
  ],
  "newEvidenceImpact": "补充信息足以改变对发布者意图的判断",
  "remainingUncertainties": ["引用原话是否完整仍需人工确认"],
  "reviewSuggestion": "建议结合引用关系改判允许"
}
```

申诉 AI 只能提供正反依据，不得直接修改最终状态。

### 4.6 `manual_reviews`

现有结构基本可以复用。统一 `final_decision`：

```text
allow
maintain_limit
need_more_context
```

当前后端使用的 `publish`、`require_edit`、`escalate` 与新版前端不一致，需要修改枚举或在接口层显式映射。

前端当前提交：

```json
{
  "finalDecision": "allow",
  "reviewReason": "补充上下文证明攻击性片段属于引用，作者实际是在反对该表达。"
}
```

`final_risk_level` 和 `correction_type` 可以改为可选，或由后端自动推导。

### 4.7 `audit_logs`

现有结构可继续使用，建议增加以下 action：

```text
content.submitted
moderation.context_built
moderation.completed
moderation.failed
evidence.validated
content.published
content.limited
content.manual_review_requested
appeal.submitted
appeal.counter_analyzed
manual_review.decided
content.restored
context.requested
```

MVP 可以不建立独立 `review_tasks` 表，继续从 `contents + appeals` 动态生成审核队列。

## 5. 统一状态与枚举

### 5.1 内容状态

| 状态 | 是否公开 | 含义 |
| --- | ---: | --- |
| pending_ai_review | 否 | AI 初审中 |
| published | 是 | 正常公开 |
| limited | 否 | 暂时限制，可申诉 |
| pending_manual_review | 否 | AI 主动转人工 |
| need_more_context | 否 | 等待用户补充上下文 |
| appeal_submitted | 否 | 已提交申诉 |
| appeal_reviewing | 否 | 反证分析或人工处理中 |
| appeal_approved | 是 | 申诉通过，恢复公开 |
| appeal_rejected | 否 | 申诉驳回 |

### 5.2 风险等级

```text
0 = L0 安全
1 = L1 低风险
2 = L2 中风险
3 = L3 高风险
```

### 5.3 风险类型

```text
insult
harassment
threat
fraud
discrimination
implicit_attack
```

### 5.4 上下文标签

```text
quote
counter_speech
report
satire
joke_uncertain
context_missing
repeated_targeting
group_pressure
```

### 5.5 处理动作

```text
allow
limit
manual_review
need_more_context
```

## 6. API 通用约定

- MVP 继续通过 `X-User-Id` Header 模拟身份；
- 数据库和 Python 使用 `snake_case`；
- HTTP JSON 使用 `camelCase`；
- 时间返回 UTC ISO 8601；
- ID 使用 UUID 字符串；
- 列表统一返回 `{ "items": [] }`；
- 错误继续使用 FastAPI `{ "detail": "..." }` 格式。

示例：

```http
X-User-Id: student_a
```

常用错误码：

| 状态码 | 场景 |
| --- | --- |
| 400 | 回复楼层不存在、业务参数错误 |
| 403 | 角色或资源权限不足 |
| 404 | 社区、话题、内容、任务不存在 |
| 409 | 重复申诉、任务已处理、状态冲突 |
| 422 | 枚举错误、复核理由为空、字段校验失败 |

### 7.7 内容审核详情

扩展现有：

```http
GET /api/contents/{contentId}/moderation
```

作者可见：

- 内容状态和用户可见理由；
- 风险类型；
- 使用的上下文；
- 可以安全公开的证据；
- 是否可申诉；
- 处理时间线。

审核员额外可见：

- 完整 AI 结果；
- `reviewerReason` 和 `rawResult`；
- `targetUserIds` 和 `uncertainties`；
- 证据校验详情；
- 模型、Prompt 和规则版本。

### 7.8 提交申诉

扩展现有：

```http
POST /api/contents/{contentId}/appeals
```

请求：

```json
{
  "appealType": "quote_or_report",
  "reason": "前半句是引用，我是在批评这种攻击表达。",
  "extraContext": "引用来自 2 楼，后半句明确表达了反对。"
}
```

响应：

```json
{
  "appealId": "appeal-001",
  "status": "submitted",
  "contentStatus": "appeal_submitted",
  "message": "申诉已提交，等待反证分析和人工复核。"

```

规则：

- 只能申诉本人内容；
- 同一内容不能存在多个进行中的申诉；
- 申诉提交后内容仍不公开；
- 申诉 AI 不能直接改变最终状态。

### 7.11 审核任务详情

```http

```

响应必须包含：

```text

evidenceValidation
appeal
counterAnalysis
timeline
```

其中 `counterAnalysis` 必须同时提供：

- 支持维持原判的依据；
- 支持改判的依据；
- 新证据影响；
- 仍未解决的不确定点；
- 给审核员的建议。

### 7.12 提交人工决定

```json

```

支持：

```text

```

```json
  "floorNumber": 4,
  "visibleToPublic": true,

```

人工允许时必须在一个业务事务中：

1. 保存 `manual_reviews`；
2. 保留原始 `moderation_records`；

5. 更新申诉状态；
6. 写入审计日志；

## 8. AI 初审与上下文契约

AI 输入至少包括：

```text
固定社区
当前话题标题
当前待审核内容
当前作者
被回复楼层
被回复用户
当前内容之前最近 5 个公开楼层
当前作者近期相关发言
目标用户近期相关发言
```

重要规则：

- 最近上下文必须限定在当前 `topic_id`；
- `parent_id` 指向的被回复内容必须单独加载；
- 连续骚扰需要检查同一作者是否多次指向同一对象；
- 对象已经要求停止，是连续骚扰的重要信号；
- 必须区分“引用并批评”和“引用后赞同攻击”。

后端固定编排：

```text
Schema 校验
→ 枚举校验
→ 必填字段校验
→ 证据 contentId 校验
→ quote 是否真实存在校验
→ 系统策略分流
→ 写入 moderation_records
→ 写入 audit_logs
```

以下情况统一转人工：

- AI 超时或返回空结果；
- 非法 JSON 或字段缺失；
- 未定义枚举；
- 引用不存在的消息；
- 证据原文无法定位；
- AI 与确定性规则冲突；
- 上下文不足。

## 9. 权限和数据可见性

后端必须保证：

1. 普通用户只能看到公开话题和公开楼层；
2. 作者可以看到自己的未公开内容；
3. 普通用户不能看到其他用户受限内容的原文；
4. 审核员可以看到全部待复核内容；
5. 只有内容作者可以发起申诉；
6. 只有审核员可以提交人工决定；
7. 普通用户只能看到 `userVisibleReason`；
8. `reviewerReason`、`rawResult` 和内部策略不能返回普通用户；
9. 等待复核和申诉中的内容不进入公开楼层；
10. 人工决定理由为空时返回 `422`；
11. 已完成任务重复提交返回 `409`；
12. 申诉成功后才允许恢复公开。

普通用户话题查询条件：

```text
visible_to_public = true
OR author_id = current_user_id
```

审核员查询不限制 `visible_to_public`。

## 10. Seed、开发顺序和联调验收

### 10.1 推荐 Seed

固定社区：


核心样例：

```text
连续骚扰：大家都看看他平时是什么样。

威胁：你放学等着，我会让你后悔。
```


### 10.4 必须跑通的三条联调链路

正常发布：

```text
提交普通内容
→ AI 建议 allow
→ 系统 publish
→ 分配楼层号
→ 其他用户可见
```

连续骚扰：

```text
同一作者连续针对同一对象
→ 对象已经要求停止
→ 新内容存在群体施压
→ pending_manual_review
→ 内容不公开且不占楼层号
→ 审核员队列可见
```

申诉改判：

```text
引用批评内容被暂时限制
→ 用户补充引用来源并申诉
→ 申诉 AI 生成正反依据
→ 审核员填写理由并改判 allow
→ appeal_approved
→ 分配最新楼层号
→ 内容恢复公开
→ 时间线保存原判、申诉和改判
```
