## 1. 本轮开发目标与范围

本轮开发以前端已经完成的交互 Demo 为产品基准，不重新设计页面和业务流程，主要目标是将原先保存在前端 `DemoContext` 中的数据结构和状态流落到真实后端与数据库。

计划补齐的核心链路为：

```text
固定社区
→ 多话题
→ 线性楼层
→ 发布前审核
→ 系统允许 / 限制 / 转人工
→ 用户查看审核详情并申诉
→ 审核员人工决定
→ 允许后恢复公开并追加楼层号
→ 保存完整处理时间线
```

本轮实际包含：

- SQLAlchemy 数据模型扩展；
- Alembic 数据库迁移；
- 社区、话题、楼层、我的发布等 API；
- 申诉和人工复核 API；
- 权限与数据可见性控制；
- 延迟楼层号分配；
- 审核记录与审计时间线；
- 可重复执行的演示 Seed；
- 自动化测试、迁移和接口冒烟验证；
- 本地运行说明更新。

本轮不包含：

- 大模型 Agent 编排；
- 申诉反证 Agent；
- OpenAI、通义、DeepSeek 等第三方 AI API；
- 正式登录、JWT、OAuth 或统一身份系统；
- Agent 和第三方 AI 相关页面数据生成。

在后续联调阶段，已将前端从本地演示数据切换为真实后端 API；`DemoContext` 只作为页面所需视图模型的适配层，不再作为业务数据源。

## 3. 实际开发过程

### 3.1 仓库与契约分析

首先阅读了前后端对接契约、现有 SQLAlchemy Model、API Router、Service、Seed、测试和前端 TypeScript 类型。

同时确认本轮只开发后端确定性业务逻辑，不扩展 Agent 和外部模型调用。

### 3.2 数据模型扩展

新增 `Topic` Model，用于表达：

```text
Scene（固定社区）
└── Topic（话题）
    └── Content（线性楼层）
```

`topics` 核心字段包括：

- `scene_id`：所属固定社区；
- `author_id`：发起人；
- `title`、`summary`、`category`；
- `status`、`visible_to_public`；
- `view_count`；
- `last_active_at`。

`contents` 新增：

- `topic_id`；
- `floor_number`；
- `target_user_id`；
- `UNIQUE(topic_id, floor_number)` 唯一约束。

`moderation_records` 新增：

- `suggested_action`；
- `system_decision`；
- `context_tags`；
- `intent`；
- `target_user_ids`；
- `context_used`；
- `uncertainties`；
- `evidence_valid`；
- `model_version`；
- `rule_version`；
- `failure_reason`。

`appeals` 新增：

- `extra_context`；
- `counter_analysis`；
- `analyzed_at`。

`manual_reviews` 调整：

- `final_risk_level` 改为可选；
- `correction_type` 改为可选；
- `final_decision` 在 API 层统一为新版枚举。

### 3.3 Alembic 迁移实现

新增迁移：

```text
backend/migrations/versions/c2417b9a6e12_add_topics_and_review_contract.py
```

迁移不仅支持空数据库，也处理旧版已有数据：

1. 创建 `topics` 表；
2. 为旧版每个已有 Scene 创建一个“历史讨论”话题；
3. 将原有 Content 回填到对应 Topic；
4. 只为原本公开的内容分配连续楼层号；
5. 受限或未公开内容保持 `floor_number = null`；
6. 增加外键、索引和唯一约束；
7. 给旧审核和申诉记录回填新字段；
8. 保留可执行的 downgrade 路径。

### 3.4 社区、话题与楼层 API

新增：

| 方法 | 路径 | 功能 |
| --- | --- | --- |
| GET | `/api/community` | 固定社区与统计数据 |
| GET | `/api/topics` | 话题列表、分类和关键词筛选 |
| POST | `/api/topics` | 创建话题及 1 楼并执行审核 |
| GET | `/api/topics/{topicId}` | 话题详情 |
| GET | `/api/topics/{topicId}/contents` | 按权限返回话题楼层 |
| POST | `/api/topics/{topicId}/contents` | 发布或回复新楼层 |
| GET | `/api/me/contents` | 当前用户的全部发布 |

创建话题时，Topic 和 1 楼 Content 在同一个业务事务中创建。1 楼通过审核后，话题才会公开；未公开话题不能继续追加楼层。

### 3.5 延迟楼层号分配

楼层号分配被集中到 `topic_service.py`：

```text
审核中 / 受限 / 待人工
→ floor_number = null

系统允许或人工允许
→ 锁定 Topic
→ 查询当前 MAX(floor_number)
→ 分配 MAX + 1
→ 更新 last_active_at
```

实现中同时使用：

- 数据库事务；
- `SELECT ... FOR UPDATE` 话题行锁；
- `UNIQUE(topic_id, floor_number)` 最终一致性保护。

因此受限内容不会提前占楼，也不会造成公开页面楼层空号。

### 3.6 审核记录和异常保护

当前仍使用确定性的 `MockAIProvider`，便于演示和测试。

后端分别保存：

```text
AI suggested_action
系统 system_decision
人工 final_decision
```

审核 Service 增加了确定性保护：

- Provider 抛出异常时不返回 500，而是转人工复核；
- Provider 返回未定义动作时转人工复核；
- 证据原文无法定位时转人工复核；
- 失败原因写入 `failure_reason`；
- 保存上下文构建、审核完成、证据校验和系统分流审计事件。

这些保护属于后端编排逻辑，不涉及第三方 AI 接入。

### 3.7 申诉和人工复核

申诉接口增加：

- 申诉类型；
- 具体理由；
- 补充上下文；
- 重复进行中申诉检查；
- 内容作者权限检查；
- 申诉审计事件。

审核任务支持：

```text
source = ai_escalation | user_appeal
status = pending | resolved
priority = normal | high
```

人工决定统一为：

```text
allow
maintain_limit
need_more_context
```

人工允许时，在同一事务中完成：

1. 保存 `manual_reviews`；
2. 保留原始 `moderation_records`；
3. 更新内容状态和公开性；
4. 分配最新楼层号；
5. 更新申诉状态；
6. 更新话题最近活跃时间；
7. 写入人工决定和恢复公开审计事件。

已完成任务再次提交决定时返回 `409`。

### 3.8 权限和数据隔离

已实现：

- 普通用户只能看到公开话题；
- 隐藏话题仅作者和审核员可访问；
- 楼层列表只返回公开楼层和当前用户自己的未公开楼层；
- 其他用户看不到受限内容原文；
- 只有作者可以申诉；
- 只有 reviewer/admin 可以查看任务和提交决定；
- 普通用户接口不返回 `reviewer_reason` 和 `raw_ai_response`；
- 审核员可以查看完整审核结果、证据、上下文和失败原因；
- 纯空白正文、申诉理由和复核理由返回 `422`。

- 请求频率限制；
- 敏感配置和密钥管理；
- 数据导出与隐私策略。

### 暂缓：Agent 与第三方 AI API

按当前项目安排暂不开发，后续接入时建议保持现有 Provider/Service 边界：

```text
Router
→ Service 编排与校验
→ Provider 接口
→ 第三方 AI API
```

需要补充的模块包括：

- 真实内容审核 Provider；
- 申诉反证 Provider/Service；
- Prompt 版本管理；
- JSON Schema 严格校验；
- 超时、重试、熔断和成本统计；
- 模型响应脱敏；
- 反证分析写入 `counter_analysis` 和 `analyzed_at`；
- Agent 只给建议，系统和人工仍掌握最终状态变更权。

## 10. 当前结论

后端和数据库已经具备支撑新版前端产品形态的主要能力，前端也已切换到真实 API。数据库结构、权限、状态流、延迟楼层号、申诉和人工复核闭环均已实现，并通过后端自动化测试和浏览器端到端人工验证。

当前项目的主要剩余工作为：

1. 在 MySQL 8.4 容器环境执行最终迁移和并发验证；
2. 将已验证的前后端闭环固化为 Playwright/组件自动化测试；
3. 补充分页、并发幂等、正式身份认证和生产安全能力。

Agent 和第三方 AI API 已按要求保持暂缓，但数据库字段和 Provider 边界已经预留，后续可以在不重写现有业务接口的前提下接入。
