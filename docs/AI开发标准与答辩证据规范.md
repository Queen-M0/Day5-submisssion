# Day5 AI 开发标准：从问题判断到答辩的全链路证据规范

> 适用项目：AI 文字内容审核与申诉复核系统  
> 适用周期：两天开发 + Day7 答辩  
> 核心原则：高分不是页面最多，而是从问题判断、方案取舍、最小原型、AI 可信使用、过程留痕到答辩抗追问，整条链路都站得住。

## 0. 本标准的目的

本文件不是功能清单，也不是让团队堆页面的开发计划。它定义的是 Day5 项目应该如何开发、如何留痕、如何证明、如何答辩。

本项目题目是“AI 文字内容审核与申诉复核系统”。它天然容易被做成一个普通敏感词过滤 Demo，但评分重点并不在“页面数量”或“按钮数量”，而在下面这些问题能否被清楚回答：

- 我们识别出的真实问题是什么？
- 为什么它不是简单敏感词过滤？
- 为什么要做上下文审核和人工复核？
- 哪些方案被比较过，为什么采纳当前方案？
- AI 在系统里做了什么，哪些地方必须保留人工判断？
- 最小原型证明了哪个核心判断？
- 哪些边界、误判和失败样例被验证过？
- 三名成员各自负责了哪些关键判断、代码和证据？
- Day7 被追问时，能否拿出文件、日志、测试和截图证明？

因此，本标准把开发拆成 7 个考点和 6 类交付证据。最终目标不是“把系统做大”，而是“把核心判断做实”。

## 2. 七个考点对应的开发标准

图片中的 7 个考点应直接变成开发过程的 7 条硬标准。

### 2.1 考点 01：识别真实问题

标准：

团队必须先写清楚题目背后的真实冲突、约束和非目标。

本题的真实问题不是：

- 做一个敏感词列表。
- 做一个聊天页面。
- 做一个管理员后台。
- 做一个大而全的社区平台。

本题的真实问题是：

- 文本是否违规高度依赖上下文。
- 引用、转述、反驳、举报、玩笑可能被误判。
- 没有敏感词的隐晦攻击、连续骚扰、外号攻击可能被漏判。
- AI 判断有不确定性，不能直接替代人工最终裁决。
- 申诉复核必须能纠正 AI 误判，而不是再次调用同一个 AI 重复错误。

必须产出：

- `docs/01-problem-framing.md`
- 其中包含：
  - 场景定义
  - 用户角色
  - 真实问题
  - 非目标
  - 评分风险
  - 本项目最小证明目标

最小可接受写法：

```markdown
## 真实问题

我们要解决的是“上下文语境中的文字风险判断”，不是关键词过滤。系统必须同时避免两类错误：

1. 误杀：用户引用、举报或反驳攻击性语言时被误封。
2. 漏判：用户通过外号、缩写、反讽和连续回复攻击某个对象时没有明显敏感词。

## 非目标

- 不做图片、语音、视频审核。
- 不做生产级处罚系统。
- 不做完全自动封禁。
- 不承诺 AI 判断 100% 正确。
- 不追求页面数量最大化。
```

### 2.2 考点 02：关键澄清问题

标准：

在开发前必须列出会改变方案的问题，并给出团队自己的假设答案。

示例澄清问题：

| 问题 | 为什么关键 | 本项目假设 |
| --- | --- | --- |
| 内容形态是论坛还是群聊？ | 决定上下文窗口和回复关系 | 采用班级群聊 + 论坛回复的混合形态 |
| 审核是在发布前还是发布后？ | 决定用户体验和状态机 | 发布前审核，高风险不公开，中风险人工复核 |
| AI 输出能否直接决定处罚？ | 决定系统可信边界 | AI 只给建议，人工复核可改判 |
| 申诉由谁处理？ | 决定闭环可信度 | 审核人员处理，不能只二次调用同一 AI |
| 敏感词表是否足够？ | 决定技术路线 | 敏感词只作为线索，核心是上下文语义判断 |
| 如何证明不是误判？ | 决定测试策略 | 准备引用、转述、反讽、连续骚扰等固定样例 |
| 如何证明个人贡献？ | 决定留痕方式 | 每人负责一类关键判断、代码模块和证据文件 |

必须产出：

- `docs/02-clarifying-questions.md`
- 每个问题必须包含：
  - 问题
  - 为什么会改变方案
  - 可选答案
  - 最终选择
  - 证据文件

### 2.3 考点 03：用证据取舍

标准：

每一个关键方案都要至少比较 2 到 3 条路线，不能只写“我们选择了某方案”。

Day5 推荐必须比较的方案：

| 决策点 | 方案 A | 方案 B | 方案 C | 推荐 |
| --- | --- | --- | --- | --- |
| 审核方式 | 纯敏感词 | 纯 AI | 规则 + AI + 人工复核 | C |
| 产品形态 | 只有用户聊天页 | 用户端 + 审核端 | 用户端 + 审核端 + 管理配置 | B 起步，C 可扩展 |
| 上下文范围 | 只看当前句 | 前后 5-10 条 | 回复链 + 前后文 + 作者历史 | C 的最小版本 |
| 申诉方式 | 用户重新发送 | 同 AI 再审一次 | 人工复核 + 留痕 | C |
| 测试方式 | 只跑成功 Demo | 成功 + 失败 | 成功 + 边界 + 误判 + 回归 | C |

必须产出：

- `docs/03-design-options.md`
- 每个方案必须写：
  - 方案说明
  - 优点
  - 缺点
  - 风险
  - 为什么选或不选
  - 对最小原型的影响

### 2.4 考点 04：驾驭 AI

标准：

AI 不是答案生成器，而是用于澄清、反证、实现和验证的协作对象。所有 AI 输出必须经过人工判断。

允许 AI 做：

- 帮助拆题。
- 帮助列风险。
- 帮助生成候选方案。
- 帮助写 Prompt 草案。
- 帮助生成测试样例。
- 帮助审查逻辑漏洞。
- 帮助写代码初稿。
- 帮助解释测试失败。

不允许 AI 直接替代：

- 最终问题定义。
- 最终方案选择。
- 最终风险等级策略。
- 最终复核结论。
- 答辩中的个人贡献说明。
- 失败样例的人工分析。

必须产出：

- `docs/ai-log.md`
- `docs/decision-log.md`
- `docs/rejected-ai-suggestions.md`

AI-log 每条记录必须包含：

```markdown
## AI-LOG-001

- 时间：
- 使用者：
- 使用目的：
- 输入摘要：
- AI 输出摘要：
- 采纳内容：
- 拒绝内容：
- 人工修改：
- 关联文件：
- 结果验证：
```

AI 使用的核心标准：

| 场景 | AI 可做 | 人必须做 |
| --- | --- | --- |
| 需求分析 | 提炼题目、列风险 | 判断真实问题和非目标 |
| 方案设计 | 生成候选方案 | 选择方案并写不选理由 |
| Prompt | 草拟结构化输出 | 确定字段、阈值和安全边界 |
| 测试 | 生成样例 | 确认样例是否符合业务语境 |
| 审核结果 | 给风险建议 | 人工复核可改判 |
| 答辩 | 帮助准备问答 | 由成员说明自己的判断和证据 |

### 2.5 考点 05：最小系统证明

标准：

两天内必须优先做一个能跑通核心判断的最小原型，而不是堆功能。

最小系统必须证明：

1. 用户提交文字内容。
2. 系统构建上下文。
3. AI 或 Mock AI 输出结构化风险判断。
4. 系统能区分发布、限制、人工复核。
5. 用户能申诉。
6. 审核人员能查看上下文、AI 理由和申诉理由。
7. 审核人员能改判或维持，并保存复核记录。

最小系统不要求：

- 完整注册登录。
- 复杂权限后台。
- 大规模数据看板。
- 多端适配完美。
- 生产级模型准确率。
- 真实封禁处罚链路。

必须产出：

- 可运行原型。
- `README.md` 一键启动说明。
- `docs/minimum-proof.md`
- `docs/demo-script.md`

最小证明句式：

```markdown
本原型证明的不是“AI 永远正确”，而是证明：

1. 审核依据可以从敏感词扩展到上下文语义；
2. 系统可以在低置信度和复杂语境下进入人工复核；
3. 用户申诉后，人工复核可以纠正 AI 误判；
4. 每次判断都有上下文、理由和记录可追溯。
```

### 2.7 考点 07：诚实的边界

标准：

答辩中必须清楚说明原型能证明什么、不能证明什么。不要把原型包装成生产系统。

必须说明：

- 当前系统只处理文字，不处理图片、语音、视频。
- 当前系统验证的是审核闭环，不承诺生产级准确率。
- 当前 AI 结果依赖 Prompt 和上下文构造，存在误判可能。
- 当前敏感词和隐晦称呼样例是演示集，不是完整风控库。
- 当前申诉复核展示流程可信性，不代表真实平台的法务和安全流程。
- 当前个人贡献用文件、提交、模块和复核记录证明，不靠口头说。

必须产出：

- `docs/day7-defense.md`
- `docs/known-limits.md`

答辩推荐表达：

```text
我们没有声称这个原型能替代真实平台审核系统。它证明的是：在文本审核这个题目里，关键不在敏感词表，而在上下文判断、申诉复核和证据留痕。我们刻意保留人工复核，是因为 AI 在引用、反讽和隐晦攻击场景中存在不确定性。这个边界是我们的设计选择，不是未完成项。
```

## 3. Day5 项目的 PRD 标准

PRD 必须回答“为什么做、给谁做、做到什么程度、怎么验收”。

推荐文件：

```text
docs/PRD.md
```

### 3.1 PRD 必备结构

```markdown
# AI 文字内容审核与申诉复核系统 PRD

## 0. 背景与一句话结论
## 1. 用户与场景
## 2. 真实用户问题
## 3. 目标与非目标
## 4. 功能范围与优先级
## 5. 核心流程
## 6. 状态定义
## 7. 风险分类与处理动作
## 8. 权限矩阵
## 9. 数据输入输出
## 10. 验收条件
## 11. 边界与异常
## 12. 成功指标
## 13. PRD 到测试的闭环要求
```

### 3.2 用户与场景标准

必须至少定义：

| 用户 | 核心诉求 | 典型场景 |
| --- | --- | --- |
| 普通用户 | 正常发言，不被误判；被限制后可申诉 | 班级群发消息、论坛回帖 |
| 审核人员 | 快速判断 AI 是否误判，保存复核结果 | 查看上下文、AI 理由、申诉理由后改判或维持 |
| 管理员 | 配置策略、查看记录和统计 | 配置风险等级、导出审核记录 |

### 3.3 目标与非目标标准

目标必须可衡量：

| ID | 目标 | 可衡量指标 |
| --- | --- | --- |
| G1 | 避免简单敏感词误杀 | 引用/举报样例至少 3 条通过 |
| G2 | 识别隐晦攻击 | 无敏感词连续攻击样例至少 2 条进入复核或限制 |
| G3 | 完成申诉闭环 | 被限制内容可申诉，审核人员可改判并保存理由 |
| G4 | 审核可解释 | 每条 AI 审核记录有风险类型、证据、上下文理由 |
| G5 | 过程可追溯 | 每个关键决策有对应文档或日志 |

非目标必须明确：

- 不做生产级处罚系统。
- 不做多模态审核。
- 不做模型训练。
- 不做完全自动封禁。
- 不把所有聊天社交功能做完。

### 3.4 功能优先级标准

P0 只保核心闭环：

| ID | 功能 | 优先级 | 验收 |
| --- | --- | --- | --- |
| F1 | 用户提交文字 | P0 | 能提交并生成内容记录 |
| F2 | 上下文聚合 | P0 | AI 输入包含当前内容、前文、回复对象 |
| F3 | AI 风险分析 | P0 | 输出结构化 JSON |
| F4 | 发布/限制/复核状态 | P0 | 根据风险等级进入正确状态 |
| F5 | 用户申诉 | P0 | 被限制内容可提交申诉理由 |
| F6 | 人工复核 | P0 | 审核人员可改判/维持并保存理由 |
| F7 | 审计记录 | P0 | 审核和复核均可追溯 |
| F8 | 失败样例库 | P0 | 有误判、漏判、边界样例 |

P1 可做但不能压过 P0：

- 数据看板
- 策略配置 UI
- 批量导出
- 更多社区场景
- 更漂亮页面

### 3.5 PRD 闭环要求

必须满足：

- 每个 P0 功能至少有 1 条验收条件。
- 每条验收条件至少有 1 条测试 case。
- 每个 AI 判断必须有用户可见理由和审核人员详细理由。
- 每次人工改判必须有原因和审计记录。
- 每个高风险结论必须能回溯到“PRD AC -> SDD 模块 -> TDD 用例 -> Demo 证据”。

## 4. Day5 项目的 SDD 标准

SDD 必须回答“系统如何设计，边界在哪里，AI 如何被约束”。

推荐文件：

```text
docs/SDD.md
```

### 4.1 SDD 必备结构

```markdown
# AI 文字内容审核与申诉复核系统 SDD

## 1. 设计目标
```


### 4.3 模块边界标准

| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| ContentService | 创建内容、更新状态、查询内容 | 不直接判断风险 |
| ContextService | 构造上下文、回复链、最近消息 | 不决定处罚 |
| RuleEngine | 敏感词、隐私格式、刷屏等确定性线索 | 不单独作为最终结论 |
| AIReviewService | 语义判断、风险分类、证据生成 | 不做最终人工裁决 |
| PolicyService | 合并规则和 AI 结果，决定状态 | 不生成 AI 理由 |
| AppealService | 申诉提交和状态流转 | 不替代审核人员复核 |
| ReviewService | 人工复核、改判、理由保存 | 不修改原始 AI 记录 |
| AuditService | 留痕 | 不参与业务判断 |

### 4.4 AI 审核契约标准

AI 输出必须是结构化 JSON，至少包含：

```json
{
  "is_violation": true,
  "risk_level": 2,
  "risk_score": 72,
  "risk_types": ["harassment", "implicit_attack"],
  "confidence": 0.79,
  "decision": "manual_review",
  "target_users": ["user_a"],
  "is_quote_or_report": false,
  "quote_context_safe": false,
  "has_implicit_attack": true,
  "has_continuous_harassment": true,
  "evidence": [
    {
      "text": "别让那个“大聪明”碰展示",
      "reason": "结合前文指向特定同学，带有反讽和排挤意味",
      "risk_type": "implicit_attack"
    }
  ],
  "context_reasoning": "连续多条消息指向同一对象，虽无显式敏感词，但构成隐晦攻击。",
  "user_visible_reason": "内容可能涉及针对他人的不友善表达，已进入人工复核。",
  "reviewer_reason": "建议审核人员重点查看最近 3 条回复和被指向对象。",
  "suggested_revision": "请聚焦具体任务分工，避免对同学进行暗示性评价。"
}
```

### 4.5 状态机标准

必须定义：

| 状态 | 含义 | 进入条件 | 下一步 |
| --- | --- | --- | --- |
| pending_ai_review | 等待 AI 审核 | 用户提交内容 | published / limited / pending_manual_review |
| published | 已公开 | 安全或低风险 | 可被后续举报 |
| limited | 已限制 | 高风险或策略限制 | 用户申诉 |
| pending_manual_review | 等待人工复核 | 中风险或低置信度 | 审核人员处理 |
| appeal_submitted | 用户已申诉 | 用户提交申诉 | 审核人员复核 |
| appeal_approved | 申诉通过 | 审核人员改判 | 内容恢复 |
| appeal_rejected | 申诉驳回 | 审核人员维持 | 保持限制 |

### 4.6 上下文构建标准

上下文不能只传当前句，至少包括：

- 当前内容。
- 作者。
- 被回复内容。
- 最近 5 到 10 条同场景消息。
- 同一作者最近相关消息。
- 是否存在引用或转述标记。
- 是否有明确攻击对象。
- 是否存在连续攻击同一对象。

上下文包格式：

```json
{
  "current": {
    "id": "msg_102",
    "author": "student_b",
    "text": "别让那个“大聪明”碰展示，懂的都懂。"
  },
  "parent": {
    "id": "msg_101",
    "author": "student_a",
    "text": "我可以负责这次展示。"
  },
  "recent_messages": [
    {
      "id": "msg_100",
      "author": "student_b",
      "text": "某些人又开始表现了。"
    }
  ],
  "signals": {
    "has_quote_marker": false,
    "has_report_intent": false,
    "same_author_repeated_target": true
  }
}
```

### 4.7 兜底标准

以下情况不能直接放行：

- AI JSON 解析失败。
- AI 置信度低。
- 上下文缺失但风险较高。
- 规则命中隐私、威胁、诈骗。
- 审核服务超时。

推荐处理：

| 异常 | 状态 | 用户提示 |
| --- | --- | --- |
| AI 超时 | pending_manual_review | 内容已进入人工复核 |
| JSON 解析失败 | pending_manual_review | 内容需要进一步确认 |
| 上下文构建失败 | pending_manual_review | 系统正在补充审核信息 |
| 高风险规则命中 | limited | 内容暂未公开，可申诉 |

## 6. AI 可信使用标准

### 6.1 AI 在产品内的可信标准

AI 审核必须符合：

- 输入可追溯：保存内容、上下文和规则线索。
- 输出结构化：不能只输出自然语言。
- 理由可解释：用户侧和审核侧分开。
- 低置信度可兜底：进入人工复核。
- 结论可申诉：不能一判到底。
- 人工可覆盖：审核人员能改判。
- 记录可复盘：保存原始 AI 输出和最终人工结论。

### 6.2 AI 在开发过程中的可信标准

开发过程必须证明：

- 哪些内容由 AI 建议。
- 哪些内容被人工采纳。
- 哪些内容被人工拒绝。
- 人工为什么修改。
- 修改后如何验证。

禁止：

- 直接把 AI 生成文档当最终文档。
- 只说“AI 帮我们完成了系统”。
- 答辩时无法说明哪部分是自己的判断。
- 没有测试就相信 AI 生成的 Prompt 或代码。

### 6.3 AI Prompt 版本标准

Prompt 需要版本化：

```text
prompts/
  moderation-v1.md
  moderation-v2-context.md
  moderation-v3-appeal-safe.md
```

每次改 Prompt 必须记录：

| 字段 | 内容 |
| --- | --- |
| 版本 | v1 / v2 / v3 |
| 修改原因 | 例如安全引用误判 |
| 修改内容 | 增加引用/举报识别字段 |
| 影响样例 | TC-QUOTE-001 |
| 回归结果 | 核心样例是否仍通过 |
| 人工判断 | 为什么接受这版 |

## 10. Day7 答辩准备标准

### 10.1 必答问题

答辩前必须准备这些问题的答案：

| 问题 | 回答要点 | 证据 |
| --- | --- | --- |
| 你们解决的真实问题是什么？ | 上下文审核，而非敏感词过滤 | `01-problem-framing.md` |
| 为什么要人工复核？ | AI 有误判风险，申诉不能重复同一错误 | `SDD.md`, `manual-review-record.md` |
| 为什么选这个方案？ | 比较纯敏感词、纯 AI、混合方案 | `03-design-options.md` |
| AI 在哪里发挥作用？ | 语义分析、证据提取、风险建议 | `prompts/`, `ai-log.md` |
| 人在哪里做判断？ | 方案取舍、Prompt 修改、复核改判 | `decision-log.md` |
| 最小原型证明了什么？ | 发布、限制、申诉、复核闭环 | `minimum-proof.md` |
| 有没有失败样例？ | 有误判、漏判、AI 失败和回归样例 | `failure-cases.md`, `TDD.md` |
| 三个人分别贡献了什么？ | 每人对应模块、判断和证据 | `contribution-map.md` |

### 10.2 答辩演示顺序

推荐 6 分钟演示：

1. 30 秒：说明真实问题。
2. 60 秒：展示方案取舍，不选纯敏感词和纯 AI。
3. 90 秒：演示正常发布和显性违规限制。
4. 90 秒：演示安全引用不误杀、隐晦攻击进复核。
5. 60 秒：演示用户申诉和审核人员改判。
6. 30 秒：展示测试、失败样例、贡献地图和边界。

### 10.3 抗追问话术

追问：为什么不用敏感词表？

回答：

```text
敏感词表只能做线索，不能做最终判断。我们准备了两个对照样例：一句包含“废物”但语境是举报，应该放行；另一句没有敏感词但连续用外号和反讽指向同一同学，应该进入复核。这说明题目核心是上下文语义审核。
```

追问：AI 判断错了怎么办？

回答：

```text
我们没有让 AI 直接成为最终裁决。低置信度、中风险和申诉场景都会进入人工复核。复核人员可以查看上下文、AI 证据和用户申诉理由，并保存改判原因。这个设计对应我们的人工复核记录和申诉测试。
```

追问：你们怎么证明不是照搬 AI？

回答：

```text
我们保留了 AI-log、decision-log 和 rejected-ai-suggestions。每条关键 AI 建议都记录了采纳、拒绝和人工修改。例如我们拒绝了“直接按敏感词拦截”的简单方案，改成规则线索 + 上下文 AI + 人工复核。
```

追问：原型不能证明真实准确率怎么办？

回答：

```text
这是我们明确承认的边界。两天原型证明的是审核闭环和关键判断，而不是生产级准确率。我们用固定成功、边界、失败和误判样例证明系统能处理题目最核心的场景，并把生产级准确率列为后续需要真实数据和人工标注评估的问题。
```

## 13. 最终提交评价标准

一个高质量提交应该让评审看到：

2. 团队没有把系统做成普通敏感词过滤器。
3. 团队比较过方案，并能解释取舍。
4. 团队知道 AI 能做什么，也知道 AI 不能替代什么。
5. 团队用最小系统证明了发布、限制、申诉、人工复核闭环。
6. 团队准备了成功、边界、失败、误判和回归样例。
7. 团队保留了 AI 使用、人工判断、踩坑修复和个人贡献证据。
8. 团队在答辩中能诚实说明原型边界。

最终一句话标准：

```text
Demo 是证据的一部分，不是全部。真正的高分来自“判断清楚、取舍清楚、原型可验证、AI 使用可信、过程可追溯、答辩能抗追问”。
```

## 14. 两天最终必须证明的六件事

图片里的核心要求可以压缩成一句话：两天最终不是证明“页面很多”，而是证明“六件事缺一不可”。

这六件事必须在文档、原型、测试和答辩中同时出现。

| 编号 | 必须证明的事 | 证明标准 | 对应文件 | 原型证据 |
| --- | --- | --- | --- | --- |
| 01 | 问题判断清楚 | 真正问题、核心冲突、非目标都能说清楚 | `docs/01-problem-framing.md`, `docs/PRD.md` | 演示同一敏感词在不同上下文下不同处理 |
| 02 | 方案取舍清楚 | 为什么选当前方案，为什么放弃其他方案 | `docs/03-design-options.md`, `docs/decision-log.md` | 展示规则、AI、人工复核三者分工 |
| 03 | AI 边界清楚 | AI 帮了什么，人判断了什么 | `docs/AI_WORKFLOW.md`, `docs/ai-log.md`, `docs/manual-review-record.md` | AI 给建议，审核人员能改判 |
| 04 | MVP 可验证 | 最小系统跑通，核心判断可演示 | `docs/minimum-proof.md`, `README.md` | 用户提交 -> 审核 -> 申诉 -> 复核 |
| 05 | 证据链可信 | 会议、分工、决策、AI、验证都落文件 | `docs/evidence-index.md`, `docs/contribution-map.md` | 每个演示 case 能找到对应测试和记录 |
| 06 | 答辩能抗追问 | 约束改变也答得上“会怎么改” | `docs/day7-defense.md`, `docs/known-limits.md` | 能说明如果接入真实平台/真实模型如何扩展 |

### 14.1 六件事的最低交付检查

每件事都必须有“结论 + 文件 + 原型 + 测试”四层证据。

| 事项 | 不能只说 | 必须给出 |
| --- | --- | --- |
| 问题判断 | “我们做内容审核” | 为什么敏感词不够、为什么上下文关键、为什么需要申诉 |
| 方案取舍 | “我们用了 AI” | 纯敏感词、纯 AI、混合方案的比较和拒绝理由 |
| AI 边界 | “AI 能审核” | AI 输入、输出、置信度、兜底、人类改判点 |
| MVP | “页面能点” | 一条主线跑通，并证明核心判断有效 |
| 证据链 | “我们开发过” | 需求、决策、日志、测试、失败、贡献文件 |
| 答辩 | “后续可以优化” | 面对约束变化的具体改法，例如换模型、加真实数据、加多级审核 |

### 14.2 六件事的答辩表达模板

```text
我们两天内不追求做完整社区平台，而是证明六件事：
第一，题目的真实问题是上下文语义审核，不是敏感词过滤；
第二，我们比较了纯敏感词、纯 AI、规则+AI+人工复核，最终选第三种；
第三，AI 只做风险建议和证据提取，最终复核保留人工判断；
第四，MVP 跑通了提交、审核、限制、申诉、复核和保存；
第五，所有关键判断都有 PRD、SDD、TDD、AI-log、failure-cases 和 review record；
第六，如果 Day7 被追问数据、模型、权限或真实场景，我们能说明当前边界和下一步改法。
```

## 15. 端、角色与团队概念对齐标准

开发前必须统一“端”和“角色”的概念，否则容易把系统边界说乱。

### 15.1 端的定义

端是技术分层，不是用户身份。

| 端 | 定义 | Day5 最低要求 |
| --- | --- | --- |
| 服务端 | 提供 API、业务规则、数据保存、AI 调用和审核状态流转 | 必须有 |
| Web 前端 | 浏览器里的用户界面，可包含多个角色视图 | 至少一个 |
| APP / 小程序客户端 | 移动端客户端 | 可不做，只在边界里说明 |
| Mock 客户端 | 用命令行、脚本或接口调试模拟前端 | 可用于缩范围，但要写进 README |

最低要求：

```text
每个项目至少两端：服务端 + 一个前端 / 客户端。
```

如果使用模拟数据、假接口或 Mock AI，必须在 `README.md` 和 `docs/known-limits.md` 写清楚：

- 哪些是真实实现。
- 哪些是 Mock。
- 为什么两天内选择 Mock。
- Mock 不影响证明的核心判断是什么。
- 后续替换成真实接口需要改哪里。

### 15.2 角色的定义

角色是同一系统里的不同使用者视角，不等于端。

Day5 推荐角色：

| 角色 | 所属端 | 核心动作 | 必须留下的证据 |
| --- | --- | --- | --- |
| 普通用户 | 用户端 / Web 前端 | 发布内容、查看审核结果、提交申诉 | `contents` 记录、申诉记录、用户侧截图 |
| 审核人员 | 审核端 / Web 前端 | 查看上下文、AI 理由、申诉理由，复核改判 | `manual_reviews` 记录、复核理由、审核端截图 |
| 管理员 | 可选管理视图 | 查看策略和记录，导出审计 | 可 P1，不做则写入非目标 |
| 系统服务 | 服务端 | 调用规则、AI、状态机、日志 | API 日志、AI 输出、状态流转记录 |

注意：

- 同一个 Web 前端可以同时承载普通用户和审核人员两个角色视图。
- 角色可以合并，但职责不能消失。
- 如果不做管理员端，必须在 PRD 非目标里说明。
- 不要因为题目说“说明系统边界”，就把所有外围功能都实现。说明边界不等于实现所有边界外功能。

### 15.3 团队职责约定

三人团队可以分工灵活，但每个人的贡献必须可追溯。

| 职责方向 | 可以由谁负责 | 不能缺失的产物 |
| --- | --- | --- |
| 产品与问题判断 | 成员 A | `01-problem-framing.md`, `PRD.md`, 演示脚本 |
| 系统设计与后端 | 成员 B | `SDD.md`, API、状态机、审核记录 |
| 测试与复核证据 | 成员 C | `TDD.md`, `failure-cases.md`, `manual-review-record.md` |

个人贡献不能只靠口头说明。每个人至少要有：

- 1 个关键判断。
- 1 份主责文档。
- 1 个核心模块或原型页面。
- 1 组测试或失败样例。
- 1 条 AI 协作记录。
- 1 条答辩可讲的取舍。

## 16. 系统原型介绍标准

系统原型介绍必须回答“这个原型证明了什么”，而不是只介绍页面。

推荐文件：

```text
docs/prototype-overview.md
```

### 16.1 原型介绍必须包含的内容

```markdown
# 系统原型说明

## 1. 原型目标
## 2. 原型不证明什么
## 3. 用户角色
## 4. 两端说明
## 5. 主流程
## 6. 核心判断点
## 7. 使用的数据
## 8. AI 在哪里参与
## 9. 人工复核在哪里参与
## 10. 演示样例
## 11. 当前 Mock 与真实实现边界
## 12. 后续替换真实能力的路径
```

### 16.2 原型目标写法

推荐写法：

```markdown
本原型用于证明“AI 文字审核不能只依赖敏感词，而应结合上下文、AI 语义判断和人工复核闭环”。原型跑通一条最小主线：

用户提交文本 -> 服务端构建上下文 -> AI 输出风险建议 -> 系统发布/限制/进入复核 -> 用户申诉 -> 审核人员查看上下文并复核 -> 保存最终结果。
```

不推荐写法：

```markdown
本系统有用户页面、审核页面、后台页面和数据看板。
```

因为这只是在列页面，不能证明核心判断。

### 16.3 原型不证明什么

必须诚实说明：

- 不证明生产级准确率。
- 不证明所有平台规则都覆盖。
- 不证明模型永远不会误判。
- 不证明真实大规模并发。
- 不证明完整处罚、账号风控和法务流程。

推荐表达：

```markdown
本原型证明审核闭环和关键判断路径，不证明生产环境的最终准确率。准确率需要真实平台数据、人工标注集、长期回归测试和多轮策略调优。
```

### 16.4 原型主流程说明

必须用一条主线介绍：

```text
普通用户端：
1. 选择班级群或论坛帖子。
2. 输入文本并发送。
3. 查看审核状态。
4. 若被限制，查看用户可见理由。
5. 提交申诉。

服务端：
1. 保存原始内容。
2. 构建上下文包。
3. 执行规则预检。
4. 调用 AI 审核。
5. 合并风险并更新状态。
6. 保存审核记录。

审核人员端：
1. 查看待复核/申诉任务。
2. 查看原文、上下文、AI 理由和用户申诉理由。
3. 选择维持、改判或要求修改。
4. 填写复核理由。
5. 保存最终结果。
```

### 16.5 原型演示必须覆盖的 5 个场景

| 场景 | 证明点 |
| --- | --- |
| 正常文本发布 | 主流程可跑通 |
| 明显攻击被限制 | 高风险能识别 |
| 引用攻击性内容但安全 | 能避免敏感词误杀 |
| 无敏感词但隐晦攻击 | 能体现上下文分析 |
| 用户申诉后人工改判 | 证明人工复核闭环 |

## 17. 原始数据设计标准

题目允许自行定义测试数据，但数据不能随手写在页面里。必须设计原始数据，并说明它为什么能证明问题。

推荐文件：

```text
docs/raw-data-design.md
data/raw/moderation_seed.json
data/raw/context_threads.json
data/processed/moderation_cases.json
```

### 17.1 原始数据和测试数据的区别

| 类型 | 说明 | 示例 |
| --- | --- | --- |
| 原始数据 | 模拟真实用户发出的消息、回复、申诉和上下文 | 群聊消息、论坛楼层、用户信息 |
| 处理后数据 | 为测试和审核引擎整理后的 case | `expected_decision`, `expected_risk_types` |
| 审核记录 | 系统运行后生成的 AI 和人工结果 | `moderation_records`, `manual_reviews` |

不要把三者混在一起。

### 17.2 原始数据必须包含的实体

```json
{
  "users": [
    {
      "id": "user_a",
      "display_name": "李同学",
      "role": "student"
    },
    {
      "id": "reviewer_1",
      "display_name": "审核老师",
      "role": "reviewer"
    }
  ],
  "scenes": [
    {
      "id": "class_group_001",
      "type": "chat_group",
      "title": "高一三班作业群"
    }
  ],
  "contents": [
    {
      "id": "msg_001",
      "scene_id": "class_group_001",
      "author_id": "user_a",
      "parent_id": null,
      "text": "我可以负责这次展示。",
      "created_at": "2026-07-16T09:00:00+08:00"
    }
  ]
}
```

### 17.3 数据集必须覆盖的语义类型

| 类型 | 最低数量 | 目的 |
| --- | ---: | --- |
| safe_normal | 2 | 证明安全内容能发布 |
| explicit_insult | 2 | 证明显性攻击能识别 |
| safe_quote_report | 2 | 证明引用/举报不误杀 |
| quote_but_attack | 1 | 证明不能用“引用”绕过 |
| implicit_attack | 2 | 证明无敏感词也能识别风险 |
| continuous_harassment | 1 | 证明连续上下文判断 |
| fraud_or_threat | 1 | 证明高风险类别 |
| appeal_false_positive | 1 | 证明申诉改判 |
| ai_failure_or_low_confidence | 1 | 证明 AI 失败兜底 |

### 17.4 原始数据设计记录模板

```markdown
## DATA-CASE-001：安全引用

- 原始消息：
- 上下文：
- 说话人：
- 被回复对象：
- 设计目的：
- 预期 AI 初判：
- 预期系统动作：
- 可能误判点：
- 对应测试用例：
- 对应演示步骤：
```

### 17.5 数据不能“赛后伪造”

所有数据样例必须在开发过程中留痕：

- 初始设计时间。
- 设计者。
- 修改记录。
- 修改原因。
- 对应测试结果。

如果某条失败样例是后期发现的，也可以加入，但必须标记为“开发中发现”，不能假装一开始就设计好了。

## 18. AI Workflow 开发标准

AI Workflow 是说明“AI 如何参与系统判断和开发过程”的文件。它比普通 Prompt 更重要，因为它定义输入、输出、状态、失败兜底和人工接管点。

推荐文件：

```text
docs/AI_WORKFLOW.md
```

### 18.1 AI Workflow 必备结构

```markdown
# AI Workflow

## 1. Workflow 目标
## 2. 输入字段
## 3. 上下文包格式
## 4. 规则预检工具
## 5. AI 审核工具
## 6. AI 输出 JSON Schema
## 7. 状态定义
## 8. 判断表
## 9. 失败兜底
## 10. 人工复核接管点
## 11. 证据记录
## 12. Demo 验证配置
```

### 18.2 输入字段标准

| 字段 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| `content_id` | string | 是 | 当前内容 ID |
| `scene_id` | string | 是 | 群聊或论坛场景 ID |
| `author_id` | string | 是 | 发送者 |
| `text` | string | 是 | 原始文本 |
| `parent_id` | string | 否 | 回复对象 |
| `recent_messages` | array | 是 | 最近上下文 |
| `author_history` | array | 否 | 作者近期相关消息 |
| `rule_hints` | array | 否 | 敏感词、URL、隐私格式等规则线索 |
| `prompt_version` | string | 是 | Prompt 版本 |

### 18.3 AI 工具定义

```markdown
### `buildContext()`

功能：
根据当前内容构造上下文包，包含当前文本、被回复内容、最近消息、作者历史和引用信号。

阻塞条件：
- 当前内容不存在。
- 场景不存在。

非阻塞条件：
- 作者历史为空。
- 被回复内容不存在但当前内容仍可审核。
```

```markdown
### `moderateText()`

功能：
调用 AI 或 Mock AI，返回结构化风险判断。

要求：
- 必须返回 JSON。
- 必须包含风险等级、风险类型、证据、用户理由和审核理由。
- 不能只返回“违规/不违规”。
```

```markdown
### `mergeDecision()`

功能：
合并规则预检、AI 判断、置信度和上下文完整度，输出最终系统动作。

动作：
- publish
- warn
- manual_review
- limit
```

### 18.4 判断表

| 条件 | 系统动作 | 原因 |
| --- | --- | --- |
| AI 安全且无规则高风险 | publish | 允许发布 |
| 命中辱骂词但 AI 判断为安全引用 | publish 或 warn | 避免误杀，但可记录 |
| AI 风险中等或置信度低 | manual_review | 不让 AI 独断 |
| AI 判断高风险且证据明确 | limit | 限制发布，可申诉 |
| AI 输出非法 JSON | manual_review | 工具失败兜底 |
| 上下文缺失且规则高风险 | manual_review | 等人工判断 |
| 威胁、诈骗、隐私泄露高置信 | limit | 高风险优先保护 |

## 19. Spec / PRD / SDD / TDD 联动开发模式

Day5 不能先写代码再补文档。推荐采用下面的顺序：

```text
Problem Framing
  -> Spec
  -> PRD
  -> Design Options
  -> SDD
  -> AI Workflow
  -> TDD
  -> MVP Implementation
  -> Test Record
  -> Failure Cases
  -> Manual Review Record
  -> Contribution Map
  -> Day7 Defense
```

### 19.1 Spec 的作用

推荐文件：

```text
docs/SPEC.md
```

Spec 负责定义系统边界和开发约束，是 PRD/SDD/TDD 的共同上游。

必须包含：

- 项目 Brief。
- 两端和角色定义。
- 输入输出。
- 数据范围。
- 非目标。
- 核心状态。
- 最小验收。
- Mock 边界。

### 19.2 PRD 的作用

PRD 负责说明“为什么做、给谁做、做到什么程度”。

必须能追溯：

```text
用户问题 -> 目标 -> 功能 -> 验收条件 -> 测试用例
```

### 19.3 SDD 的作用

SDD 负责说明“系统怎么实现，以及为什么这样分模块”。

必须能追溯：

```text
验收条件 -> 模块 -> 数据模型 -> API -> 状态机 -> 兜底策略
```

### 19.4 TDD 的作用

TDD 负责说明“怎么证明判断有效”。

必须能追溯：

```text
验收条件 -> 测试用例 -> 固定输入 -> 期望输出 -> 实际结果 -> 是否回归
```

### 19.5 AI Workflow 的作用

AI Workflow 负责说明“AI 怎么被调用、被限制、被记录和被人工接管”。

必须能追溯：

```text
当前文本 -> 上下文包 -> 规则线索 -> AI 输出 -> 合并决策 -> 人工复核 -> 最终状态
```

### 19.6 开发时的文件写入顺序

| 阶段 | 先写文件 | 再做动作 | 结束后补充 |
| --- | --- | --- | --- |
| 需求澄清 | `01-problem-framing.md`, `02-clarifying-questions.md` | 团队讨论 | `decision-log.md` |
| 方案选择 | `03-design-options.md` | 确定 MVP 范围 | `rejected-ai-suggestions.md` |
| 需求规格 | `SPEC.md`, `PRD.md` | 拆验收条件 | `evidence-index.md` |
| 系统设计 | `SDD.md`, `AI_WORKFLOW.md` | 设计接口和状态机 | `known-limits.md` |
| 测试设计 | `TDD.md`, `moderation-cases.json` | 写测试/样例 | `test-record.md` |
| 实现 | 代码与 Prompt | 跑主流程 | `ai-log.md`, `failure-cases.md` |
| 复核 | `manual-review-record.md` | 演示改判 | `audit-log` 或截图 |
| 答辩 | `day7-defense.md` | 预演追问 | 更新 `contribution-map.md` |

## 20. 六类 AI 协作记录标准

图片要求“六类工作，至少五类有真实记录”。Day5 建议六类都覆盖；如果时间不够，至少覆盖五类，但 **反证** 和 **Review** 不能缺。反证用于证明团队主动挑战了自己的方案，Review 用于证明团队审查过证据链、答辩漏洞和系统边界。

推荐文件：

```text
docs/ai-collaboration-record.md
```

### 20.1 六类 AI 工作定义

| 类别 | AI 做什么 | 人做什么 | Day5 示例 |
| --- | --- | --- | --- |
| 澄清 | 发现信息缺口或隐含约束 | 判断哪些问题真的改变方案 | AI 提出“发布前还是发布后审核”，团队决定发布前审核 |
| 对比 | 生成或比较不同路线 | 选择路线并写不选理由 | 比较纯敏感词、纯 AI、混合方案 |
| 反证 | 挑战假设、找错误或反例 | 接受有效反例，拒绝不符合场景的反例 | AI 找到“引用后赞同攻击”会绕过简单引用规则 |
| 实现 | 辅助代码、界面、接口、规则 | 审查代码并跑测试 | 生成审核 JSON Schema 或状态机代码 |
| 验证 | 生成或改进测试与边界样例 | 固定样例并写预期 | 生成 TC-QUOTE、TC-IMPLICIT、TC-FAIL |
| Review | 审查方案、证据链和答辩漏洞 | 选择要修的问题和保留边界 | AI 提醒缺少个人贡献证据，团队补 `contribution-map.md` |

### 20.2 每条关键记录必须包含

```markdown
## AI-COLLAB-001

- 类别：澄清 / 对比 / 反证 / 实现 / 验证 / Review
- 目标：
- 输入摘要：
- AI 输出摘要：
- 人工采纳：
- 人工拒绝：
- 人工修改：
- 实际影响：
- 关联文件：
- 验证结果：
```

### 20.3 最低数量要求

| 要求 | 标准 |
| --- | --- |
| 六类工作 | 至少五类有真实记录，建议六类都有 |
| 反证记录 | 必须至少 1 条，且要包含假设、反例、是否采纳和实际修改 |
| Review 记录 | 必须至少 1 条，且要包含审查对象、发现问题、人工判断和修复结果 |
| 被拒绝或明显修改的 AI 建议 | 至少 2 条 |
| 每条关键记录 | 必须包含目标、输入、输出摘要、采纳/拒绝、人工判断、实际影响 |
| 禁止 | 赛后伪造日志，或把 AI 输出直接当最终裁决 |

### 20.4 被拒绝 AI 建议示例

```markdown
## AI-COLLAB-004

- 类别：对比
- 目标：选择审核策略
- AI 输出摘要：建议先用敏感词表快速拦截，再做申诉。
- 人工采纳：保留“敏感词作为线索”。
- 人工拒绝：拒绝“命中敏感词直接拦截”，因为题目明确要求避免引用、转述、反讽误判。
- 人工修改：改为“规则预检 + 上下文 AI + 人工复核”。
- 实际影响：SDD 中新增 RuleEngine、AIReviewService、ReviewService 三段式设计。
- 关联文件：`docs/03-design-options.md`, `docs/SDD.md`
- 验证结果：TC-QUOTE-001 可通过安全引用样例。
```

## 24. 可测试的边界不是“边界”

用户特别提醒：“可以测试的边界 ==> 边界”。这里要纠正一个常见误区：边界不是为了凑测试用例临时编出来的“可测试边界”，而是系统、业务、数据、AI 和人工职责的真实边界。测试只是验证边界有没有被守住。

### 24.1 边界的定义

边界必须回答：

- 系统做什么，不做什么。
- AI 判断到哪里为止，人从哪里接管。
- 原型证明什么，不证明什么。
- 数据覆盖哪些场景，不覆盖哪些真实风险。
- Mock 能替代什么，不能替代什么。
- 当前两天 MVP 的结论能推广到哪里，不能推广到哪里。

### 24.2 Day5 必须写清的 6 类边界

| 边界类型 | 必须写清 | 对应文件 |
| --- | --- | --- |
| 题目边界 | 只处理文字，不处理图片、语音、视频；只做审核闭环，不做生产处罚系统 | `SPEC.md`, `PRD.md` |
| 产品边界 | 至少服务端 + 一个前端；管理员配置可后置；不堆页面 | `prototype-overview.md`, `README.md` |
| 数据边界 | 使用自定义演示数据；不代表真实平台全量风险分布 | `raw-data-design.md`, `TDD.md` |
| AI 边界 | AI 给建议和证据，不直接做最终裁决 | `AI_WORKFLOW.md`, `SDD.md` |
| 人工边界 | 审核人员处理申诉和低置信度，不负责替模型背书 | `manual-review-record.md` |
| 结论边界 | MVP 证明链路可行，不证明生产级准确率 | `known-limits.md`, `day7-defense.md` |

### 24.3 边界到测试的转换

边界先写在文档里，再转成测试。

| 边界 | 测试方式 | 期望 |
| --- | --- | --- |
| AI 不能直接最终裁决 | 中风险样例 | 进入 `pending_manual_review` |
| 引用不应被敏感词误杀 | 安全引用样例 | `published` 或 `warn` |
| 引用不能成为攻击借口 | 引用后赞同攻击样例 | `limited` |
| 无敏感词也可能违规 | 隐晦攻击样例 | `manual_review` |
| AI 输出失败不能放行 | 非法 JSON / 超时样例 | `manual_review` |
| MVP 不证明生产准确率 | 答辩说明 | 写入 `known-limits.md` |

| AI 使用质量 | 是否会采纳、拒绝、修改 AI 建议 | `ai-collaboration-record.md` |
| 协作贡献 | 是否帮助团队补链路、修风险 | `decision-log.md`, Review 记录 |
| 答辩解释能力 | 是否能解释自己负责部分 | `day7-defense.md` 个人问答 |

### 26.3 缺材料的计分方式

缺某项材料通常只在对应评分维度中扣分，不应被重复扣分。但如果缺材料导致结论无法复核，会影响多个维度。

示例：

- 只有 Demo，没有过程证据：MVP 维度可能有分，但问题判断、方案取舍、AI 使用、团队协作会失分。
- 有 AI-log，但与真实实现明显不一致：AI 使用质量和过程可信度都失分。
- 成员无法解释自己负责模块：个人贡献分进入复核。

## 27. 红线：触线即返工或进入复核

以下情况属于基础交付红线。触发后要么返工，要么进入复核；不能靠答辩临场解释糊过去。

| 编号 | 红线 | 影响 | 修复要求 |
| --- | --- | --- | --- |
| 01 | 没有可运行或可交互系统 | MVP 不能证明 | 至少跑通一条主路径 |
| 02 | 未满足题目明确的技术端与角色要求 | 系统边界不成立 | README 和 SDD 补清端、角色、服务端 |
| 03 | 原型无法演示题目中的核心判断 | Demo 失去证明力 | 补安全引用、隐晦攻击、申诉复核场景 |
| 04 | 只有最终 Demo，没有真实过程证据 | 结论无法复核 | 补 PRD/SDD/TDD/AI-log/test-record/decision-log |
| 05 | AI 日志或测试记录与实际过程明显不一致 | 过程可信度失败 | 重新标注真实记录，承认缺口 |
| 06 | 团队成员无法解释自己负责的部分 | 个人贡献进入复核 | 补 contribution-map，并由本人准备问答 |
| 07 | 伪造过程、验证结果或个人贡献证据 | 严重违规 | 不允许；只能保留真实记录和真实边界 |

### 27.1 不允许的做法

- 赛后伪造 AI 日志。
- 把没有跑过的测试写成通过。
- 把 AI 输出直接当最终裁决，却说有人审查过。
- 把别人的贡献写成自己的。
- 临场新增没有证据的说法。
- 用 PPT 替代真实文件和测试记录。

### 27.2 允许但必须诚实标注的做法

- 使用 Mock AI，但要说明接口契约和替换路径。
- 使用模拟数据，但要说明数据设计目的和覆盖范围。
- 某些功能未完成，但要写入非目标或已知边界。
- 某些测试失败，但要记录根因和后续修复计划。
- 某些 AI 建议被拒绝，但要说明拒绝理由。

## 28. AI 日志和测试记录必须与真实实现一致

AI 日志和测试记录不是装饰材料。它们必须能对应真实代码、真实数据、真实运行结果。

### 28.1 AI 日志一致性检查

每条 AI 日志至少要能反查到：

```text
AI 使用目的 -> 输入摘要 -> AI 输出摘要 -> 人工判断 -> 影响文件 -> 验证结果
```

检查表：

- [ ] AI 建议影响了哪个文件。
- [ ] 人工采纳了什么。
- [ ] 人工拒绝了什么。
- [ ] 拒绝或修改的理由是什么。
- [ ] 修改后是否运行测试或做人工验证。
- [ ] 记录时间与开发过程大体一致。

### 28.2 测试记录一致性检查

每条测试记录至少要能反查到：

```text
测试用例 -> 固定输入 -> 预期输出 -> 实际输出 -> 运行命令/截图 -> 结论
```

测试记录必须包含：

- 正常用例。
- 边界用例。
- 失败用例。
- 误判/漏判用例。
- 申诉复核用例。
- 回归用例。

如果某条测试没有真正跑，只能写成“计划测试”或“未执行”，不能写成 PASS。

### 28.3 不一致时怎么处理

如果发现日志和实现不一致，不要硬圆。

推荐处理：

```markdown
## INCONSISTENCY-001

- 发现时间：
- 不一致内容：
- 实际情况：
- 影响范围：
- 修正方式：
- 是否影响答辩结论：
- 后续计划：
```

诚实修正比伪造完整更安全。

## 29. 反证与 Review 强制记录

反证和 Review 是本项目必须有的两类记录，不是可选加分项。

### 29.1 反证记录标准

反证不是“找茬”，而是主动找能推翻方案的例子。

推荐文件：

```text
docs/counter-evidence.md
```

模板：

```markdown
## COUNTER-001：安全引用规则可能被绕过

- 原始假设：
- AI 或成员提出的反例：
- 反例输入：
- 反例为什么能推翻假设：
- 团队判断：
- 采纳/拒绝：
- 方案修改：
- 对应测试：
- 最终结果：
```

Day5 内容审核项目推荐至少准备 4 条反证：

| 反证 | 推翻的简单假设 |
| --- | --- |
| 引用后赞同辱骂 | “有引用标记就安全” |
| 无敏感词连续反讽 | “没有敏感词就安全” |
| 低置信度高风险 | “AI 给了结果就能发布” |
| 申诉再次调用同一 AI | “申诉可以自动复判” |

### 29.2 Review 记录标准

Review 是审查方案、证据链和答辩漏洞。

推荐文件：

```text
docs/review-record.md
```

模板：

```markdown
## REVIEW-001：Day6 证据链审查

- 审查时间：
- 审查人：
- 审查对象：
- 发现问题：
- 严重程度：
- 人工判断：
- 修复动作：
- 修复后证据：
- 是否影响答辩：
```

至少做 3 类 Review：

| Review 类型 | 审查对象 | 目标 |
| --- | --- | --- |
| 方案 Review | `03-design-options.md`, `SDD.md` | 检查取舍是否能解释 |
| 证据 Review | `evidence-index.md`, `test-record.md` | 检查观点是否能反查 |
| 答辩 Review | `day7-defense.md` | 检查追问是否答得上 |

## 30. Day7：六分钟展示，六分钟追问

Day7 答辩建议按“六分钟展示，六分钟追问”准备。展示不要塞满功能，追问不要临场新增无证据说法。

### 30.1 六分钟展示结构

| 时间 | 内容 | 必讲点 |
| --- | --- | --- |
| 60s | 问题诊断 | 真正问题、核心冲突、非目标 |
| 90s | 方案取舍 | 3 个方案，为什么选最终方案 |
| 90s | 端到端设计 | 各端、数据流、AI 位置 |
| 120s | MVP 演示 | 主路径、规则触发、一个边界/失败 |
| 60s | 验证与风险 | 测试结果、失败场景、当前不能证明什么 |
| 60s | 个人贡献 | 每人一句关键贡献和对应证据 |

### 30.2 六分钟追问准备

每个人都要准备自己负责部分的追问：

| 追问方向 | 谁答 | 必须带证据 |
| --- | --- | --- |
| 为什么这是核心问题 | PRD 负责人 | `01-problem-framing.md` |
| 为什么选混合方案 | 方案负责人 | `03-design-options.md` |
| AI 具体做了什么 | AI Workflow 负责人 | `AI_WORKFLOW.md`, `ai-log.md` |
| 怎么证明没误判 | 测试负责人 | `TDD.md`, `test-record.md` |
| 失败样例是什么 | 测试/Review 负责人 | `failure-cases.md`, `counter-evidence.md` |
| 个人做了什么 | 每个人 | `contribution-map.md` |

### 30.3 答辩提醒

提前想好：

- 约束改变会怎么改。
- 如果接入真实数据会怎么扩展。
- 如果 AI 误判会怎么处理。
- 如果审核人员不一致会怎么处理。
- 如果测试失败，当前不能证明什么。

不要：

- 一个人回答所有问题。
- 临场新增没有文件支撑的说法。
- 把 Demo 当作全部证据。
- 回避失败样例。
- 声称原型能证明生产级准确率。

## 31. “比的不是完整”最终原则

这类题比的不是系统是否完整，而是判断、取舍、过程证据和诚实边界。

题目给出的线索只是起点。每组应该主动补充至少 3 个更隐蔽的失败样例，主动收窄范围，不丢分。

Day5 内容审核项目推荐补充的隐蔽失败样例：

| 隐蔽失败样例 | 检查什么 |
| --- | --- |
| 同一句“你真行”在友好玩笑和讽刺攻击中不同处理 | 上下文语气 |
| 用户复制别人辱骂内容并附加“说得对” | 引用不等于安全 |
| 多人轮流对同一对象阴阳怪气 | 群体骚扰 |
| 只用外号不点名，但上下文对象明确 | 指向性 |
| AI 置信度低但规则命中高风险 | 兜底策略 |
| 用户申诉理由很短但上下文支持 | 人工复核价值 |

最终评价一句话：

```text
把一个真实例子从头到尾演示清楚，比堆十个页面更有说服力。高分来自清楚判断、清楚取舍、真实过程证据和诚实边界。
```
