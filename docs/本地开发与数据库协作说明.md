# ContextGuard 本地开发与数据库协作说明

## 1. 数据库协作原则

团队不提交 MySQL 数据目录，也不互相复制本地数据库。Git 中只同步：

1. `backend/migrations/` 中的 Alembic 表结构迁移；
2. `backend/app/seed/` 中可重复执行的 Demo 种子数据；
3. `.env.example` 中不含密钥的配置模板。

每位开发者拥有独立的本地 MySQL 容器。运行时产生的内容、申诉和审核记录只存在于各自环境；需要共享实时数据时，另外部署一套团队 dev 数据库，连接信息放本地 `.env`，不得提交 Git。

## 2. 第一次启动 MySQL

在项目根目录执行：

```bash
cd backend
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements-dev.txt
cd ..
./scripts/bootstrap_db.sh
cd backend
uvicorn app.main:app --reload
```

初始化脚本会启动并等待 MySQL 健康、按需创建本地 `backend/.env`、执行 `alembic upgrade head`、运行幂等 Seed，并执行 `alembic check`。数据库结构以 `backend/migrations/versions/` 中的迁移历史为准，不提交 MySQL 数据卷或个人 `.env`。

API 地址为 `http://127.0.0.1:8000`，交互文档为 `http://127.0.0.1:8000/docs`。

不使用 Docker 时，可把 `.env` 中的 `DATABASE_URL` 改为 `sqlite:///./contextguard.db`，其余命令不变。

## 3. 启动前端

另开终端执行：

```bash
cd frontend
npm install
npm run dev
```

页面地址为 `http://localhost:5173`。Vite 会把 `/api` 代理到本地 FastAPI。

## 4. 拉取他人数据库变更

```bash
git pull
cd backend
source .venv/bin/activate
alembic upgrade head
python -m app.seed.seed_demo
```

Seed 使用固定 ID 并检查现有记录，可重复执行，不会重复插入演示数据。

## 5. 提交表结构变更

修改 SQLAlchemy Model 后，由负责该变更的开发者生成并提交迁移：

```bash
cd backend
alembic revision --autogenerate -m "add policy rules"
alembic upgrade head
```

提交前必须检查迁移文件并在空数据库执行一次 `alembic upgrade head`。同一时间尽量只由一人生成迁移；若两个分支都产生了 Alembic head，合并时使用 `alembic merge heads` 明确合并，不要手工删除另一方迁移。

## 6. 分支边界

当前后端已提供社区、话题、线性楼层、申诉、人工复核、审核历史和时间线。前端 `DemoContext` 已改为真实 API 的页面适配层，业务数据和写操作均进入后端数据库。

Agent 与第三方 AI API 不在当前实现范围内。后续 Agent 工作建议分别从最新 `main` 拉功能分支：

- 上下文初审 Agent：扩展 `backend/app/providers/` 与 `backend/app/services/moderation_service.py`；
- 申诉反证 Agent：新增独立 Provider/Service，由人工复核接口编排；
- 其他扩展模块：通过新 Router/Service 接入，避免把模型调用写进 API 路由。
