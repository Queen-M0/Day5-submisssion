# 0. 背景与一句话结论

## 0.1 项目背景

社区中的文字内容是否存在辱骂、骚扰、歧视、诈骗或威胁，通常不能只根据某一个词判断。

同一句话在不同上下文中，可能具有完全不同的含义：

- 用户引用攻击性内容进行批评，不应被当作攻击者；
- 用户没有使用明显敏感词，但连续针对同一对象发布隐晦内容，仍可能构成骚扰；
- 用户使用反讽、外号、暗示或群体施压，单句看似正常，结合历史内容后风险明显；
- AI 可能误解引用关系、说话对象或真实意图，不能直接承担最终申诉裁决。

因此，本项目不做简单的敏感词过滤器，而是构建一个完整的内容审核闭环：

```text
内容提交
→ 上下文聚合
→ AI 风险分析
→ 证据真实性校验
→ 允许 / 限制 / 人工复核
→ 用户申诉
→ AI 申诉反证
→ 审核员最终裁决
→ 全流程留痕
```

## 0.2 一句话结论

> 言鉴 AI 是一个以 AI 上下文审核为核心的极简文字讨论社区。每一楼内容在公开前都经过 AI 风险分析和证据校验；AI 不确定或用户申诉时，由审核员完成最终裁决，系统保存原判、申诉、改判和理由。

## 0.3 本原型要证明什么

本原型证明的不是“AI 永远正确”，而是：

1. 审核依据可以从敏感词扩展到上下文语义；
2. AI 可以识别引用、对象、意图和连续行为；
3. AI 输出可以被结构化约束和证据校验；
4. AI 不确定或发生异常时，系统可以安全转人工；
5. 用户可以申诉，人工可以纠正 AI 误判；
6. 每一次判断都有输入、证据、理由和状态变化可追溯。

# 1. 用户与场景

## 1.1 使用场景

本项目采用一个固定的文字讨论社区，社区内包含多个话题。

每个话题采用线性楼层结构：

```text
社区
└── 话题
    ├── 1 楼：发起话题
    ├── 2 楼：参与讨论
    ├── 3 楼：参与讨论
    └── 4 楼：回复某楼，但仍作为新楼层展示
```

用户可以：

- 发起一个新话题；
- 在已有话题中发布新楼层；
- 选择回复某一个楼层；
- 查看自己的所有发布及审核状态；
- 对被限制的内容发起申诉。

系统不实现嵌套评论。用户回复某一楼时，回复内容仍作为最新楼层出现，并保存被回复楼层和目标用户。

## 1.2 用户角色

### 普通用户

核心诉求：

- 正常内容能够顺利公开；
- 引用、批评、举报等安全语境不被误判；
- 被限制时能够看到具体原因；
- 能够补充上下文并发起申诉；
- 能够查看申诉进度和最终结果。

典型操作：

- 浏览社区话题；
- 发起新话题；
- 发布新楼层；
- 回复指定楼层；
- 查看“我的发布”；
- 查看 AI 审核详情；
- 提交申诉。

### 审核员

核心诉求：

- 快速识别 AI 是否误判；
- 能同时查看当前内容、上下文、证据和申诉；
- 能够维持原判或改判；
- 每次人工决定都有理由和记录。

典型操作：

- 查看 AI 主动转人工的内容；
- 查看用户申诉；
- 查看原始上下文；
- 查看第一次 AI 判断和证据校验；
- 查看申诉反证 AI 的正反依据；
- 允许发布、维持限制或要求补充信息。

### 系统服务

系统服务不是前端用户角色，负责：

- 聚合上下文；
- 调用 AI；
- 校验 AI 输出；
- 执行状态分流；
- 保存审核、申诉、人工复核和审计记录。

# 2. 真实用户问题

## 2.1 核心问题

### 问题 1：安全内容可能被简单规则误杀

例如：

> 他说“你就是个废物”，这种攻击别人的表达很不合适。

虽然出现攻击性词语，但发布者是在引用并批评该表达。

### 问题 2：隐晦骚扰可能被漏判

例如某用户连续发布：

```text
某些人又开始装了。
每次都是他拖后腿。
大家都看看他平时是什么样。
```

单条内容可能没有明显敏感词，但结合发布者、目标对象和连续行为，可能构成骚扰或群体施压。

### 问题 3：AI 判断可能缺乏真实证据

AI 可能：

- 引用输入中不存在的句子；
- 识别错说话者；
- 识别错被回复对象；
- 在上下文不足时仍然给出强结论；
- 返回无法解析的格式。

### 问题 4：申诉可能重复第一次 AI 错误

如果申诉只是把同样的内容再次交给同一个 AI，系统无法证明申诉具有纠错能力。

### 问题 5：用户不知道内容为什么没有公开

只有“审核不通过”无法支持申诉，也无法建立用户信任。

## 2.2 核心冲突

| 目标 | 冲突 |
| --- | --- |
| 快速自动审核 | AI 可能误判 |
| 保护社区成员 | 不能过度限制正常表达 |
| 给出详细原因 | 不能向恶意用户泄露完整审核策略 |
| 使用上下文 | 不能无限加载整个话题和全部历史 |
| 支持申诉 | 不能让相同 AI 重复裁决 |
| 原型范围可控 | 仍需真实跑通完整闭环 |

## 2.3 当前产品假设

1. 系统在内容公开前进行审核；
2. 只有最终允许的内容进入公开社区；
3. 等待复核、受限和申诉中的内容只对作者和审核员可见；
4. AI 只提供分析和建议，不直接承担申诉最终裁决；
5. 当前只处理纯文字；
6. 当前只有一个固定社区；
7. 每次审核只加载与当前内容最相关的有限上下文；
8. 申诉允许用户补充说明和上下文；
9. 审核员作出最终决定时必须填写理由。

# 3. 目标与非目标

## 3.1 产品目标

| ID | 目标 | 可衡量指标 |
| --- | --- | --- |
| G1 | 跑通文字发布审核闭环 | 用户提交后可进入公开、限制或人工复核状态 |
| G2 | 避免简单敏感词误杀 | 至少 3 条引用、批评或举报样例获得正确处理 |
| G3 | 识别无敏感词的隐晦风险 | 至少 2 条连续骚扰样例进入限制或人工复核 |
| G4 | 审核结果可解释 | 每条演示审核记录包含风险类型、证据、上下文依据和不确定点 |
| G5 | AI 输出可校验 | 演示案例中的证据均可定位到原始内容或上下文 |
| G6 | AI 失败可降级 | 模型超时、非法输出和证据无效均进入人工复核 |
| G7 | 完成申诉闭环 | 被限制内容可申诉，审核员可维持或改判并保存理由 |
| G8 | 结果真实影响公开状态 | 审核通过后进入楼层，限制时不进入，申诉成功后恢复公开 |
| G9 | 全流程可追溯 | 提交、AI 审核、系统分流、申诉和人工决定形成时间线 |
| G10 | 两人贡献可复核 | 两名成员均有 AI、前端、后端、测试和决策证据 |

## 3.2 MVP 成功定义

MVP 完成必须同时满足：

1. 一个固定社区可访问；
2. 社区中可查看多个话题；
3. 用户可发起话题和发布新楼层；
4. 用户可回复指定楼层，但不产生嵌套评论；
5. 每一楼公开前经过 AI 审核；
6. 审核结果包含允许、限制、人工复核三类；
7. 审核通过后分配公开楼层号；
8. 用户可在“我的发布”查看所有状态；
9. 被限制内容可提交申诉；
10. 审核员可查看上下文、AI 依据和申诉；
11. 审核员可维持或改判并填写理由；
12. 申诉成功后内容进入公开楼层；
13. 完整流程有时间线；
14. 至少完成一条端到端正向案例和一条反向案例；
15. 至少完成一次申诉改判。

## 3.3 非目标

本项目当前不做：

- 多社区创建和管理；
- 加入或退出社区；
- 社区成员申请、邀请、踢出、禁言；
- 社区管理员权限体系；
- 完整注册登录；
- 点赞、收藏、关注、私信；
- 热榜、推荐流和消息通知；
- 嵌套评论；
- 内容编辑和删除；
- 图片、视频、语音审核；
- 生产级处罚或封禁系统；
- 模型训练和微调；
- 大规模并发；
- 完整运营数据看板；
- 生产级准确率承诺。

# 5. 核心流程

## 5.1 发起新话题

```text
用户填写标题和 1 楼正文
→ PENDING_AI_REVIEW
→ 构建上下文
→ AI 初审
→ 证据校验
→ 系统分流
```

- 允许：创建并公开话题；
- 人工复核：话题不公开，作者和审核员可见；
- 限制：话题不公开，作者可申诉。

## 5.2 发布新楼层

```text
用户在话题中输入内容
→ 可选回复某一楼
→ 暂不公开
→ AI 审核
→ 审核通过后追加到公开楼层
```

## 5.3 回复指定楼层

- 保存被回复内容 ID；
- 保存目标用户 ID；
- 新内容仍成为最新楼层；
- 页面显示“回复 X 楼 @用户”；
- 不生成嵌套评论树。

## 5.4 AI 初审

AI 输入至少包含：

- 当前内容；
- 当前作者；
- 所属话题标题；
- 被回复楼层；
- 当前楼层前最近 5 楼；
- 当前作者近期相关内容；
- 目标用户近期相关内容；
- 引用、连续行为等信号。

系统随后执行：

- Schema 校验；
- 枚举校验；
- 证据定位校验；
- 必填字段校验；
- 失败降级。

## 5.5 系统分流

### 允许发布

- 安全内容；
- 明确的安全引用、批评、举报或规则讨论。

### 暂时限制

- 明确威胁；
- 明确诈骗；
- 严重攻击且证据充分。

### 人工复核

- 中风险；
- 上下文不足；
- 反讽、玩笑或关系不明；
- AI 输出异常；
- 证据无法验证；
- 用户申诉。

## 5.6 用户申诉

1. 查看用户可见审核理由；
2. 填写申诉理由；
3. 补充上下文；
4. 提交申诉；
5. 内容进入申诉处理中；
6. 其他普通用户仍不可见。

## 5.7 AI 申诉反证

读取：

- 第一次 AI 分析；
- 第一次系统决定；
- 原始上下文；
- 用户申诉理由；
- 用户补充上下文。

输出：

- 原判可能成立的依据；
- 原判可能错误的依据；
- 新信息是否足以改变判断；
- 未解决的不确定点；
- 给审核员的复核建议。

AI 不直接输出最终申诉裁决。

## 5.8 人工复核

审核员必须查看：

- 当前内容；
- 所属话题；
- 被回复楼层；
- 相关上下文；
- 第一次 AI 分析；
- 证据校验；
- 用户申诉；
- AI 反证结果。

审核员可以：

- 允许发布；
- 维持限制；
- 要求补充上下文。

提交前必须填写理由。

## 5.9 改判恢复

申诉通过后：

- 内容状态改为最终允许；
- 分配当前最新楼层号；
- 内容公开进入话题；
- 时间线记录改判人、理由和时间。

# 6. 状态定义

## 6.1 内容状态

| 状态 | 含义 | 是否公开 | 下一步 |
| --- | --- | ---: | --- |
| DRAFT | 用户尚未提交 | 否 | 提交 |
| PENDING_AI_REVIEW | 等待或正在 AI 初审 | 否 | 发布、限制或人工复核 |
| PUBLISHED | 已公开 | 是 | 正常展示 |
| LIMITED | 暂时限制 | 否 | 用户申诉 |
| PENDING_MANUAL_REVIEW | 等待人工复核 | 否 | 审核员处理 |
| NEED_MORE_CONTEXT | 需要补充信息 | 否 | 用户补充 |
| APPEAL_SUBMITTED | 用户已申诉 | 否 | AI 反证 |
| APPEAL_ANALYZING | 申诉 AI 分析中 | 否 | 人工复核 |
| APPEAL_REVIEWING | 审核员处理中 | 否 | 允许或驳回 |
| APPEAL_APPROVED | 申诉通过 | 是 | 分配楼层号并公开 |
| APPEAL_REJECTED | 申诉驳回 | 否 | 维持限制 |

## 6.2 楼层号规则

- 审核中不分配公开楼层号；
- 审核通过后分配当前话题的最新楼层号；
- 被限制内容不会在公开楼层中留下空号；
- 申诉成功后按恢复公开时的顺序追加为最新楼层；
- 内部使用唯一内容 ID 追踪原始提交顺序。

## 6.3 可见性规则

| 状态 | 作者 | 其他用户 | 审核员 |
| --- | ---: | ---: | ---: |
| PENDING_AI_REVIEW | 可见 | 不可见 | 可见 |
| PUBLISHED | 可见 | 可见 | 可见 |
| LIMITED | 可见 | 不可见 | 可见 |
| PENDING_MANUAL_REVIEW | 可见 | 不可见 | 可见 |
| NEED_MORE_CONTEXT | 可见 | 不可见 | 可见 |
| APPEAL_SUBMITTED | 可见 | 不可见 | 可见 |
| APPEAL_ANALYZING | 可见 | 不可见 | 可见 |
| APPEAL_REVIEWING | 可见 | 不可见 | 可见 |
| APPEAL_APPROVED | 可见 | 可见 | 可见 |
| APPEAL_REJECTED | 可见 | 不可见 | 可见 |

# 7. 风险分类与处理动作

## 7.1 MVP 风险类型

| 类型 | 含义 |
| --- | --- |
| SAFE | 未发现足以限制的风险 |
| INSULT | 辱骂或人身攻击 |
| HARASSMENT | 骚扰或持续性攻击 |
| THREAT | 威胁或恐吓 |
| FRAUD | 诈骗或诱导 |
| DISCRIMINATION | 歧视性攻击 |

辅助语境标签：

- QUOTE
- COUNTER_SPEECH
- REPORT
- SATIRE
- JOKE_UNCERTAIN
- CONTEXT_MISSING
- REPEATED_TARGETING
- GROUP_PRESSURE

## 7.2 风险等级

| 等级 | 含义 | 默认动作 |
| --- | --- | --- |
| LOW | 安全或轻微风险 | 允许发布 |
| MEDIUM | 存在争议或不确定性 | 人工复核 |
| HIGH | 明确且严重风险 | 暂时限制，允许申诉 |

## 7.3 处理动作

- ALLOW：允许公开；
- LIMIT：暂时限制；
- MANUAL_REVIEW：转人工复核；
- NEED_MORE_CONTEXT：要求补充上下文。

## 7.4 AI 建议与系统决定分离

系统分别保存：

- AI 建议动作；
- 系统最终分流动作；
- 人工最终决定。

置信度不能单独作为自动限制依据。

# 10. 验收条件

## 10.1 社区与发布

### AC-001：浏览公开社区
只展示 PUBLISHED 或 APPEAL_APPROVED 的话题和楼层。

### AC-002：发起话题
1 楼审核通过后话题才公开；1 楼受限时其他用户不可见。

### AC-003：发布新楼层
审核通过后成为当前最新楼层，审核前其他用户不可见。

### AC-004：回复指定楼层
记录被回复内容和目标用户，新内容仍线性展示。

### AC-005：楼层号无公开空缺
审核通过或申诉成功后才分配公开楼层号。

## 10.2 AI 审核

### AC-006：上下文构建
AI 输入至少包含当前内容、作者、话题、被回复楼层、最近 5 楼和目标用户。

### AC-007：结构化输出
结果必须包含风险等级、风险类别、证据、上下文依据、不确定点和建议动作。

### AC-008：安全引用不被简单误杀
引用攻击性内容进行批评、举报或反驳时，不能只因敏感词自动限制。

### AC-009：连续骚扰使用上下文判断
同一用户连续针对同一对象，且对象已要求停止时，应限制或转人工。

### AC-010：证据必须真实存在
AI 证据必须能在当前内容或上下文中定位。

### AC-011：AI 异常安全降级
超时、空结果、非法 JSON、字段缺失、证据无效均转人工复核。

### AC-012：AI 建议不是最终裁决
分别保存 AI 建议、系统分流和人工最终决定。

## 10.3 我的发布和申诉

### AC-013：我的发布完整可见
作者可查看自己的审核中、已公开、受限、复核中和申诉结果。

### AC-014：限制内容可申诉
LIMITED 内容可填写申诉理由和补充上下文。

### AC-015：申诉不能由相同 AI 直接终审
申诉 AI 只能提供反证，最终结果由审核员提交。

### AC-016：人工改判必须填写理由
理由为空时禁止提交。

### AC-017：申诉成功恢复公开
改判允许后分配最新楼层号并公开。

### AC-018：申诉驳回维持限制
内容继续仅对作者和审核员可见。

## 10.4 审计与追溯

### AC-019：完整时间线
记录用户提交、上下文构建、AI 初审、证据校验、分流、申诉、反证和人工决定。

### AC-020：关键版本留痕
P1 完成时记录模型、Prompt 和规则版本。

# 11. 边界与异常

## 11.1 输入边界

- 话题标题 1–80 字；
- 楼层内容 1–2000 字；
- 空白或纯标点禁止提交；
- 上下文最多加载最近 5 楼；
- 用户只能回复当前话题中存在的公开楼层。

## 11.2 AI 异常处理

| 异常 | 系统处理 |
| --- | --- |
| AI 超时 | 转人工复核 |
| AI 返回空内容 | 转人工复核 |
| JSON 解析失败 | 转人工复核 |
| 字段缺失 | 转人工复核 |
| 证据不存在 | 转人工复核并记录 |
| 上下文加载失败 | 转人工复核 |
| 模型结果分歧 | 转人工复核 |

## 11.3 业务异常

- 1 楼未通过审核：整个话题不公开；
- 被回复楼层不存在：禁止提交；
- 同一内容只允许一个进行中的申诉；
- 申诉无新信息：允许提交，审核员可维持原判；
- 审核员不填理由：禁止提交；
- 内容恢复公开时：追加为当前最新楼层；
- 数据保存失败：不改变公开状态并记录错误。

## 11.4 诚实边界

本项目不承诺：

- 开放世界生产级准确率；
- 覆盖全部网络语言和隐晦表达；
- 完全替代真实平台审核员；
- 所有模型和 Prompt 下结果一致。

# 18. 已确认决定与暂定项

## 18.1 已确认决定

- 社区场景；
- 单社区、多话题；
- 线性楼层式讨论；
- 支持回复指定楼层；
- 不做嵌套评论；
- 不做点赞、收藏、关注；
- 每一楼公开前审核；
- 只公开最终允许的内容；
- 用户有“我的发布”；
- 支持用户申诉；
- 支持人工复核；
- 两人按完整业务链纵向拆分；
- 两人各自使用本地 MySQL；
- 使用迁移文件和 seed 数据保持一致；
- 先完成技术骨架，再冻结文档和契约，再进行正式开发。
