# 言鉴 AI：P0 + P1 完整功能实施过程与汇报说明

> 项目：ContextGuard（言鉴 AI）
> 完成时间：2026-07-17
> 本轮范围：完成全部 P0、原 P1 高分增强，并将“数据统计看板、审核规则配置”由 P2 提升到 P1；P2 其余功能不做。

---

## 1. 最终完成了什么

系统现在形成一条可运行、可追溯的完整链路：

```text
账号登录
→ 用户发话题/楼层
→ 初审 Agent 构建上下文并分析
→ 主模型与辅助模型独立复核
→ 证据真实性校验
→ 版本化规则引擎分流
→ 发布 / 限制 / 转人工
→ 用户查看原因并申诉
→ 申诉反证 Agent 挑战原判
→ 审核员人工终审
→ 状态、版本、证据和审计时间线全部留存
→ 统计看板汇总运营指标
```

最终功能包括：

1. 账号密码登录、令牌恢复、退出和角色权限；
2. 固定社区、多话题、线性楼层、回复指定楼层；
3. MiMo 上下文初审 Agent；
4. 消息 ID 级证据绑定与证据真实性校验；
5. MiMo 申诉反证 Agent；
6. 主辅双模型独立复核和分歧检测；
7. AI 建议、规则分流、人工决定三段式对比；
8. Prompt、模型、规则版本留痕；
9. 人工复核、补充上下文、申诉改判和时间线；
10. 数据统计看板；
11. 审核规则配置与版本历史；
12. 一键测试案例、状态筛选和话题搜索。

## 2. 实施过程

### 2.1 第一阶段：先冻结业务契约

开发前先把系统中最容易前后端不一致的部分固定下来：

- 内容状态：`published`、`limited`、`pending_manual_review`、申诉相关状态；
- AI 建议：`allow`、`limit`、`manual_review`；
- 系统决定：`publish`、`limit`、`manual_review`；
- 人工决定：`allow`、`maintain_limit`、`need_more_context`；
- 风险等级：L0–L3；
- 风险类型：侮辱、骚扰、威胁、欺诈、歧视、隐性攻击；
- Evidence 必须包含真实 `contentId`、逐字证据、理由和风险类型。

这样做的目的，是让“模型说了什么”“系统做了什么”“人工最后怎么判”成为三个不同概念，避免把 AI 输出直接当最终处罚。

### 2.2 第二阶段：完成 P0 主闭环

先实现用户提交内容后的基本状态机：

```text
提交内容（默认不可见、无公开楼层号）
→ 加载话题、回复对象、最近 5 楼、作者历史、目标用户历史
→ 调用初审 Agent
→ Pydantic 严格 Schema 校验
→ 校验证据是否逐字存在于输入
→ 分流
```

分流规则：

- 安全：公开，并分配当前最新楼层号；
- 高风险且证据充分：限制，不占公开楼层号；
- 不确定、证据无效、模型异常：转人工，不占公开楼层号。

随后补齐用户申诉：受限内容作者可以填写申诉类型、申诉理由和补充上下文。申诉反证 Agent 不负责直接改判，而是同时输出：

- 支持维持原判的依据；
- 支持改判的依据；
- 新证据对原判的影响；
- 仍然存在的不确定点；
- 给人工审核员的建议。

审核员最终选择允许、维持限制或要求补充，并必须填写具体理由。原始 AI 记录不会被覆盖。

### 2.3 第三阶段：接入真实 MiMo

Provider 使用 MiMo 的 OpenAI-compatible `chat/completions` 协议。密钥仅从 `backend/.env` 读取，未写入 Git、前端或文档。

当前模型配置：

| 用途 | 模型 | Prompt |
| --- | --- | --- |
| 初审主模型 | `mimo-v2.5` | `moderation-v2` |
| 初审辅助模型 | `mimo-v2.5-pro` | `moderation-v2` |
| 申诉反证 | 当前主 Provider | `appeal-critic-v1` |

模型输出必须先通过 JSON 解析、字段完整性、枚举、数值范围和 Evidence 校验。任何一层失败都转人工。

真实联调曾发现辅助模型把 `riskScore` 输出成 `0.1`，而契约要求 0–100 整数。系统没有偷偷放宽校验，而是安全转人工；随后将 Prompt 升级到 `moderation-v2`，明确：

- `riskLevel` 是 0–3 整数；
- `riskScore` 是 0–100 整数；
- `confidence` 才是 0–1 小数。

这段过程可以用于汇报“发现问题—保守降级—升级 Prompt—保留严格契约”的工程闭环。

### 2.4 第四阶段：双模型分歧检测

双模型不是投票系统，也不是两个模型谁票多听谁的。它的目标是发现不确定性。

```text
主模型 mimo-v2.5 独立分析
→ 主证据校验

辅助模型 mimo-v2.5-pro 独立分析
→ 辅助证据校验

比较两路结果
→ 一致：交给规则引擎正常分流
→ 分歧/辅助调用失败：按配置转人工
```

以下情况认定为分歧：

1. 两路建议动作不同；
2. 风险等级相差 2 级或以上；
3. 两路都判断高风险，但风险类型没有交集；
4. 辅助模型证据无法在真实输入中定位；
5. 辅助模型超时、HTTP 异常、非法 JSON 或 Schema 不匹配。

保存内容包括主辅模型名、Prompt 版本、辅助决定、风险等级、风险类型、证据、证据有效性、原始结构化结果、分歧原因、系统处理和失败原因。

### 2.5 第五阶段：版本化规则引擎

新增规则配置后，没有让页面只做“保存表单”，而是真正接入每次审核：

```text
AI 原始结果
→ 证据校验
→ 读取当前 active 规则版本
→ 应用安全阈值
→ 得到 systemDecision
```

规则优先级：

1. 证据真实性未通过且配置要求证据：转人工；
2. 风险类型未允许自动处置：转人工；
3. 置信度低于阈值：转人工；
4. 双模型分歧且配置要求分歧转人工：转人工；
5. 证据充分且达到自动限制等级：限制；
6. 达到人工复核等级但模型建议公开：转人工；
7. 其余情况采用模型建议。

安全兜底条件优先级最高，不能被后续自动限制或发布规则覆盖。

每次保存规则执行以下动作：

1. 校验人工复核阈值不能高于自动限制阈值；
2. 将旧规则标记为 `is_active=false`；
3. 创建新的 `community-vN`；
4. 保存修改理由和修改人；
5. 写入 `moderation_rules.updated` 审计日志；
6. 后续新内容立即使用新版本；
7. 旧审核记录继续保留原 `rule_version`。

### 2.6 第六阶段：登录和权限

用户表新增 `password_hash`。密码使用 PBKDF2-SHA256、随机盐和 210,000 次迭代保存，不保存明文。

登录过程：

```text
用户名 + 密码
→ 查询用户
→ 恒定时间比较密码哈希
→ 签发 12 小时 HMAC 签名令牌
→ 前端保存令牌
→ Axios 自动附加 Authorization: Bearer
→ 后端验签和过期时间
→ 根据 user/reviewer/admin 做权限控制
```

前端增加路由守卫：未登录只能看到登录页；刷新页面会校验并恢复令牌；退出后清除本地令牌。审核统计、规则配置和人工复核接口只有 reviewer/admin 可访问。

说明：这是比赛演示所需的轻量登录，不等同于生产环境完整 IAM。生产环境还应增加刷新令牌、撤销列表、登录限流、验证码、密钥轮换和 HTTPS-only Cookie。

### 2.7 第七阶段：数据统计看板

统计页展示：

- 内容总量、公开量、待人工复核量、受限量；
- 申诉总数、待处理申诉、申诉通过率；
- 人工复核数、人工改判数、改判率；
- 双模型比较数、分歧数、分歧率；
- L0–L3 风险等级分布；
- publish / warn / manual_review / limit 系统分流分布；
- 最近 7 天内容提交量和人工复核量；
- 当前规则版本、主模型、辅助模型和双模型开关。

统计口径：

```text
申诉通过率 = approved / (approved + rejected) × 100%
人工改判率 = 人工最终决定与原 systemDecision 不同的数量 / 人工复核数 × 100%
双模型分歧率 = divergent comparison / comparison 总数 × 100%
风险与分流分布 = 每条内容最新一条 moderation_record
```

分母为 0 时返回 0，避免页面出现 NaN。

### 2.8 第八阶段：前端展示补齐

审核员菜单新增：

- 待复核队列；
- 复核记录；
- 数据统计看板；
- 审核规则配置。

页面补强：

- 待复核队列支持“模型分歧”筛选并显示分歧数；
- 审核详情显示主辅模型结果、分歧原因、模型失败和证据有效性；
- 用户发布结果和“我的发布”审核详情显示双模型结果；
- 复核记录显示“AI 原始建议 → 规则系统分流 → 人工最终决定”；
- 一键案例扩充为正常交流、安全引用、连续骚扰、明确威胁、低置信度边界、群体施压和反欺诈提醒；
- 登录后根据角色进入用户端或审核员端。

## 3. 数据库设计

项目通过 Alembic 管理迁移。当前核心表如下。

### 3.1 `users`

保存账号、展示名、角色和密码哈希。

关键字段：`id`、`username`、`display_name`、`role`、`password_hash`、`created_at`。

### 3.2 `scenes`

保存社区场景。当前演示固定单社区，但数据层保留 scene 概念。

### 3.3 `topics`

保存话题标题、分类、作者、状态、公开性、浏览数和最后活跃时间。1 楼未审核通过时整个话题不公开。

### 3.4 `contents`

保存话题楼层正文、作者、回复目标、楼层号、状态和公开性。

重要设计：未公开内容 `floor_number` 为空，审核通过后再分配最新公开楼层，避免公开页面出现空号。

### 3.5 `moderation_records`

保存每次主模型初审的完整记录：

- provider、model_version、prompt_version、rule_version；
- risk_level、risk_score、risk_types、confidence；
- 模型原始 decision、suggested_action、system_decision；
- evidence、evidence_valid；
- 上下文标签、意图、目标用户、使用的上下文、不确定点；
- 用户理由、审核员理由、原始结构化输出和失败原因。

它保证模型原始建议和系统最终分流不会互相覆盖。

### 3.6 `moderation_comparisons`

保存双模型比较，一条主审核记录最多对应一条 comparison。

关键字段：`primary_record_id`、辅助模型信息、辅助结果、证据有效性、`divergent`、`divergence_reasons`、`system_resolution`、`failure_reason`。

### 3.7 `moderation_rule_configs`

保存规则版本历史。关键字段：

- `version`、`name`；
- `enabled_risk_types`；
- `auto_limit_min_risk_level`；
- `manual_review_min_risk_level`；
- `min_confidence`；
- `require_grounded_evidence`；
- `route_divergence_to_manual`；
- `is_active`、`change_reason`、`updated_by`。

### 3.8 `appeals`

保存申诉类型、理由、补充上下文、当前反证摘要和申诉状态。

### 3.9 `appeal_analysis_records`

每调用一次申诉反证 Agent 就新增一条，不覆盖旧结果。保存 provider、模型、Prompt、结构化分析、证据有效性和失败原因。

### 3.10 `manual_reviews`

保存人工复核的原系统决定、最终决定、最终风险等级、具体理由、纠错类型和审核员。

### 3.11 `audit_logs`

保存内容提交、上下文构建、AI 审核、证据校验、双模型比较、分歧、规则更新、申诉、补充上下文和人工决定等事件。

## 6. 推荐演示流程

### 第一步：登录与角色

1. 打开 `http://127.0.0.1:5173`；
2. 使用 `zhangsan / user123` 登录；
3. 说明未登录不能访问业务页面，普通用户看不到审核员菜单。

### 第二步：正常内容和上下文审核

1. 进入一个话题；
2. 一键加载“正常交流”或“安全引用”；
3. 提交后展示 AI 建议、系统分流、双模型结果、证据和上下文。

### 第三步：边界内容转人工

1. 一键加载“群体施压”或“低置信度边界”；
2. 展示内容没有公开楼层号；
3. 说明系统在证据不足、低置信度或模型分歧时不会武断处罚，而是转人工。

### 第四步：申诉反证

1. 在“我的发布”中打开受限内容；
2. 提交“这是引用/台词”的补充上下文；
3. 展示反证 Agent 同时列出支持原判和支持改判的依据。

### 第五步：审核员终审

1. 退出并使用 `reviewer / review123` 登录；
2. 在待复核队列筛选“模型分歧”；
3. 打开任务，查看原始上下文、主辅模型、证据、申诉反证；
4. 填写人工理由并作出最终决定；
5. 在复核记录展示三段式对比和人工改判标签。

### 第六步：P1 运营能力

1. 打开数据统计看板，解释申诉通过率、改判率和分歧率；
2. 打开审核规则配置；
3. 调高最低置信度并填写修改理由；
4. 保存生成新 `community-vN`；
5. 展示历史版本仍在，后续审核记录使用新规则版本。

## 7. 可直接用于汇报的话术

> 我们不是做一个输入文本就返回“违规/不违规”的模型壳，而是实现了一个可追溯的人机协同审核系统。初审 Agent 会读取话题、回复对象、最近楼层和双方历史，并要求每条证据绑定真实消息 ID。主模型和辅助模型独立分析，不做简单投票；一旦动作、风险等级、风险类型或证据出现明显分歧，就交给人工。系统还在模型之后增加版本化规则引擎，所以模型建议、系统分流和人工终审是三层分离的。用户可以申诉，反证 Agent 会主动挑战第一次判断，但最终裁决仍由审核员完成。所有模型、Prompt、规则、证据、状态变化和人工理由都会留痕。P1 还补充了规则配置和数据统计，让项目从“能跑的 Demo”变成了具备基础运营能力的审核系统。

## 8. 范围边界

本轮明确不做 P2 其余功能：

- 审核记录导出；
- 多社区；
- 用户管理后台；
- 批量审核；
- 消息通知。

系统也不宣称生产级模型准确率。当前登录是演示级轻量实现；真实上线还需要完整 IAM、安全审计、限流、监控、密钥轮换和更大规模评测。
