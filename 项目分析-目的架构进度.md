## 一、项目目的

**言鉴 AI 是一个 AI 文字内容审核与申诉复核社区**——面向"文字论坛"场景的内容治理系统。

它要解决的不是"有没有敏感词"，而是**有上下文的语义判断**：

- 区分"引用批评"与"本人攻击"
- 识别没有敏感词的连续骚扰
- 在误判时给用户一条**可申诉、可改判、全程可追溯**的通道

核心设计理念三条：

1. **AI 负责语义判断，代码负责确定性校验与状态流转，人工保留最终裁决权**
2. **AI 建议、系统分流、人工结果必须分层保存，原判不可被覆盖**
3. **模型失败不得导致高风险内容自动公开**（超时/非法 JSON/证据失效 → 统一转人工）

项目定位：2 人、Day5–6 开发、Day7 答辩的课程项目；约束是"两天内可交付、可运行、可演示、可追溯"。

## 二、项目架构

### 2.1 技术选型（方案 C，五维评分 4.65 胜出）

```
React + Vite 前端
  + FastAPI 模块化单体后端
  + MySQL 8.4（Alembic 迁移 + 幂等 Seed）
  + 受控 AI 审核工作流 + 人工复核
```

不选纯规则（无法理解语义）、不选单模型自动裁决（证据不可信、申诉易重复错误）、不选多 Agent 委员会（两天内不可控）。

### 2.2 分层结构

| 层 | 技术 | 职责 |
|---|---|---|
| 前端 | React + Vite | 社区/话题、我的发布、审核详情/申诉、审核员工作台；`DemoContext` 已改为真实 API 适配层 |
| 后端 API | FastAPI | community / topics / contents / appeals / reviewer / me / auth 路由 |
| 后端 Service | 模块化单体 | context（上下文构建）、moderation（初审编排）、topic（楼层号）、appeal、review、audit、serializers |
| AI 能力 | Provider 适配层 | 当前 `MockAIProvider`；真实 AI Provider 边界已预留；后续接入 Appeal Critic 反证 |
| 数据层 | MySQL + Alembic + Seed | 8 张核心表；追加式 `audit_logs`；幂等演示数据 |

### 2.3 核心数据模型

```
users → scenes（固定社区）→ topics（话题）→ contents（线性楼层）
                                          ↓
                              moderation_records（AI+系统分层结果）
                                          ↓
                                   appeals（申诉+反证）
                                          ↓
                                manual_reviews（人工终判）
                                          ↓
                                  audit_logs（时间线）
```

`contents` 关键约束：`UNIQUE(topic_id, floor_number)`，仅公开时在事务中分配楼层号，受限内容不占楼、不空号。

### 2.4 审核与申诉闭环

```
用户提交 → 构建上下文 + AI 初审 + 证据校验（代码编排）
        → 系统策略路由
            ├ allow      → 公开 + 分配楼层号
            ├ limit      → 用户申诉 → 人工复核改判
            └ manual     → 人工复核（allow / maintain_limit / need_more_context）
```

设计硬性要求：AI 输出必须 Schema 校验 + 证据（`contentId`/`quote` 可定位）校验，失败统一转人工；人工结果独立成表，不物理覆盖原始 AI 记录。
