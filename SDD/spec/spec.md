# 言鉴 AI（ContextGuard）项目 SPEC

| 项目项 | 内容 |
| --- | --- |
| 题目 | AI 文字内容审核与申诉复核系统 |
| 产品形态 | 单社区、多话题、线性楼层式文字社区 |
| 团队规模 | 2 人 |
| 周期 | Day 5–Day 6 开发，Day 7 答辩 |
| 文档性质 | 开发前需求与验收基线 |
| 上游材料 | 课堂题面、团队澄清讨论 |
| 下游材料 | PRD、Design、Dev Workflow、Test Strategy、QA Gates |

---

## 1. 问题定义

系统要解决的不是“文本中是否出现敏感词”，而是：

> 如何结合说话者、被回复对象、上下文、引用关系和历史行为，对社区文字内容作出可解释、可申诉、可人工纠正的风险判断。

系统必须同时降低两类错误：

1. **误杀**：用户引用、批评、举报攻击性表达时，被错误识别为攻击者；
2. **漏判**：用户不使用明显敏感词，却通过外号、反讽、连续针对和群体施压实施骚扰。

---

## 2. 用户与角色

### 2.1 普通用户

能够：

- 浏览公开社区与话题；
- 发起话题；
- 发布新楼层；
- 回复指定楼层，但回复仍线性展示；
- 查看自己的所有发布状态；
- 查看用户可见审核理由；
- 对受限内容申诉并补充上下文。

### 2.2 审核员

能够：

- 查看 AI 主动转人工的内容；
- 查看用户申诉；
- 查看当前内容、回复关系、上下文、AI 证据和不确定点；
- 查看申诉反证分析；
- 允许发布、维持限制或要求补充上下文；
- 填写具体理由并留下审计记录。

### 2.3 系统服务

负责：

- 聚合上下文；
- 调用 AI；
- 校验结构化输出和证据真实性；
- 执行允许、限制、人工复核分流；
- 保存申诉、人工决定和时间线。

---

## 3. 核心业务规则

1. 每条内容在公开前进行审核；
2. 审核通过后才分配公开楼层号；
3. 审核中、受限、待人工、申诉中的内容只对作者和审核员可见；
4. 用户可回复指定楼层，但不形成嵌套评论树；
5. AI 建议、系统分流和人工最终决定必须分层保存；
6. AI 证据必须可以定位到当前内容或提供的上下文；
7. AI 超时、非法 JSON、字段缺失和无效证据统一转人工；
8. 申诉不得只让相同 AI 对相同输入重复裁决；
9. 申诉 AI 只提供支持维持与支持改判的依据；
10. 人工最终决定必须填写理由；
11. 原始 AI 记录不得被人工结果覆盖；
12. 改判允许后，内容作为当前最新公开楼层恢复。

---

## 4. MVP 范围

### 4.1 必须实现

- 固定社区；
- 多个话题；
- 发起话题；
- 发布楼层；
- 回复指定楼层；
- “我的发布”；
- AI 上下文初审；
- 证据真实性校验；
- 允许、限制、人工复核三类分流；
- 审核详情；
- 用户申诉；
- AI 申诉反证；
- 审核员待复核列表与详情；
- 人工维持或改判；
- 审计时间线；
- AI 失败降级。

### 4.2 明确不做

- 多社区创建、加入和成员管理；
- 完整注册登录；
- 点赞、收藏、关注、私信；
- 嵌套评论；
- 内容图片、音频、视频审核；
- 生产级处罚系统；
- 自主多 Agent 委员会；
- RAG、向量数据库；
- Kafka、RabbitMQ 等消息中间件；
- 生产级准确率承诺。

---

## 5. AI 边界

### AI 负责

- 识别说话者、目标对象和意图；
- 识别引用、转述、举报、反驳和赞同攻击；
- 判断辱骂、骚扰、威胁、诈骗、歧视等风险；
- 分析连续针对和群体施压；
- 输出证据、上下文依据、不确定点与建议动作；
- 在申诉阶段主动寻找原判断漏洞。

### 确定性代码负责

- 输入校验；
- 权限校验；
- 上下文查询；
- Schema 校验；
- 证据定位；
- 状态机和楼层号事务；
- 失败降级；
- 审计留痕。

### 人工负责

- 争议内容和申诉的最终裁决；
- 判断是否采纳 AI 建议；
- 填写改判或维持理由；
- 对最终结果负责。

---

## 6. 核心验收

| ID | 验收要求 |
| --- | --- |
| S-AC-01 | 正常内容审核后公开并分配楼层号 |
| S-AC-02 | 受限内容不进入公开楼层 |
| S-AC-03 | 安全引用不能仅因敏感词被自动限制 |
| S-AC-04 | 无敏感词的连续骚扰能进入限制或人工复核 |
| S-AC-05 | AI 编造证据时转人工并记录失败原因 |
| S-AC-06 | AI 超时或非法输出时不自动放行 |
| S-AC-07 | 作者可对受限内容提交申诉和补充上下文 |
| S-AC-08 | 申诉 AI 同时给出维持与改判依据，不直接终审 |
| S-AC-09 | 审核员不填理由时不能提交决定 |
| S-AC-10 | 人工改判允许后内容恢复公开 |
| S-AC-11 | 原判、申诉、反证、人工决定均可在时间线追溯 |
| S-AC-12 | 至少一条完整 E2E 和三条自补失败/边界样例通过验证 |

---

## 7. 证据要求

每一个关键结论应能指向至少一种证据：

- 需求或设计文件；
- 原型页面；
- Git commit；
- Issue 或会议记录；
- AI 协作日志；
- Prompt 快照；
- 测试命令和结果；
- 截图或录屏；
- 人工 Review 记录。

不得将赛后补写的推测描述成开发当时已经发生的事实。
