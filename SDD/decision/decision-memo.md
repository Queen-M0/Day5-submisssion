# 最终决策备忘录

## 1. 决策主题

为“AI 文字内容审核与申诉复核系统”选择两天内能够交付、验证和答辩的产品与技术方案。

## 2. 候选方案

### 方案 A：规则主导审核

依赖敏感词、正则、固定阈值和行为计数。

优点：

- 实现快；
- 确定性强；
- 易测试。

关键缺点：

- 容易误杀引用和举报；
- 容易漏判反讽、外号和隐晦骚扰；
- 不能体现题目的上下文判断核心。

### 方案 B：单模型直接裁决

将当前内容和上下文一次性交给大模型，由模型直接返回允许、限制或转人工。

优点：

- 语义能力强；
- 接入快；
- 演示直观。

关键缺点：

- 可能编造证据；
- 输出格式和结果不稳定；
- 置信度不能直接当作真实正确率；
- 申诉容易重复第一次错误；
- 模型直接限制会模糊责任边界。

### 方案 C：受控 AI 工作流 + 证据校验 + 人工复核

固定流程：

```text
上下文构建
→ 初审 AI
→ Schema 校验
→ 证据校验
→ 系统分流
→ 用户申诉
→ 申诉反证 AI
→ 人工最终裁决
→ 审计时间线
```

优点：

- AI 负责复杂语义；
- 代码负责确定性检查；
- 人工负责争议终审；
- 能处理引用误判、连续骚扰、AI 失败和申诉改判；
- 结果可追溯。

代价：

- 模块和状态较多；
- 需要额外证据校验和人工复核页面；
- 前后端契约必须尽早冻结。

## 3. 最终决定

> 选择方案 C：React Web 前端 + FastAPI 模块化单体 + MySQL + 受控 AI 工作流 + 人工复核。

## 4. 产品形态决定

- 一个固定社区；
- 多个话题；
- 线性楼层；
- 可以回复指定楼层；
- 不做嵌套评论；
- 不做多社区、点赞、收藏和完整登录；
- 普通用户和审核员使用同一个 Web 工程中的不同角色视图。

## 5. 选择依据

| 维度 | 规则方案 | 单模型方案 | 受控工作流 |
| --- | ---: | ---: | ---: |
| 上下文问题覆盖 | 2 | 4 | 5 |
| 两天交付可行性 | 5 | 4 | 4 |
| AI 可信性 | 4 | 2 | 5 |
| 答辩展示 | 2 | 4 | 5 |
| 申诉纠错 | 1 | 2 | 5 |

受控工作流不是代码最少的方案，但它是唯一能同时证明：

- AI 真正使用上下文；
- AI 证据可验证；
- 系统不会盲信 AI；
- 申诉能引入新信息；
- 人工能够纠正误判；
- 原判和改判均可追溯。

## 6. 放弃项

本期放弃：

- 多社区；
- 社区加入和人员管理；
- 嵌套评论；
- 风险分数超过阈值直接封禁；
- 申诉时重复调用同一 AI 并直接改判；
- 自主多 Agent 审核委员会；
- RAG；
- 微服务；
- 消息队列；
- 生产级认证系统。

## 7. 决策风险与缓解

| 风险 | 缓解 |
| --- | --- |
| 模型无法接通 | 使用相同 Schema 的 Mock Provider，仅作为失败兜底 |
| 模型超时 | 转人工，不自动放行 |
| AI 编造证据 | 证据定位失败后转人工 |
| 前后端进度不同 | 先冻结 Schema，使用同类型 Mock Adapter |
| 状态复杂 | 使用统一枚举和状态机测试 |
| 楼层号并发 | 最终公开事务中分配，并设置唯一约束 |
| 时间不足 | 冻结 P1，优先正常、风险、申诉改判三条链 |

## 8. 变更规则

实施阶段若要推翻以下决定，必须记录原因：

- 产品形态；
- AI、人和代码的责任边界；
- 公开前审核；
- 申诉人工终审；
- 证据校验；
- 楼层号分配规则；
- P0 与非目标范围。

变更记录应写入 `DOCS/collaboration/decision-change-log.md`。
