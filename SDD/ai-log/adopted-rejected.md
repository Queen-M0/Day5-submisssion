# AI 建议采纳、拒绝与人工修改清单

## 1. 采纳的关键建议

| ID | AI 建议 | 人工判断 | 采纳后的影响 |
| --- | --- | --- | --- |
| A-01 | 用极简社区承载审核，而不是单句表单 | 能真实展示审核结果是否进入公开区 | 增加社区、话题和楼层 |
| A-02 | 回复采用线性楼层，不做嵌套评论 | 保留上下文，范围可控 | 保存 replyTo，作为最新楼层展示 |
| A-03 | 使用受控 AI 工作流 | AI、代码、人工边界更清晰 | 增加 Schema、证据校验和 Policy Router |
| A-04 | 申诉使用独立反证角色 | 能主动发现原判漏洞 | 增加 Appeal Critic |
| A-05 | AI 失败统一转人工 | 同时避免默认放行和默认误伤 | 增加失败状态和人工队列 |
| A-06 | 两人各自使用本地 MySQL | 不依赖一人电脑，便于并行 | 使用 Alembic 与 Seed 同步 |

## 2. 被拒绝或明显修改的建议

### R-01：做多个社区、加入社区和成员管理

- 原建议：建设多个社区，支持加入、退出和社区人员。
- 决定：拒绝。
- 原因：
  - 与 AI 审核核心关系弱；
  - 增加权限和 CRUD；
  - 两天内会稀释主链。
- 替代：一个固定社区，多个话题。

### R-02：实现传统帖子评论树

- 原建议：在帖子下实现评论和评论回复。
- 决定：明显修改。
- 原因：
  - 树形关系和嵌套 UI 成本高；
  - 不利于快速加载统一上下文。
- 替代：回复指定楼层，但新内容仍作为最新楼层。

### R-03：风险分数超过 80 自动限制或封禁

- 原建议：使用模型风险分数直接决定高风险处罚。
- 决定：拒绝。
- 原因：
  - 模型分数不是经过校准的真实概率；
  - 可能忽略引用和证据有效性；
  - 责任边界不清。
- 替代：分数仅用于辅助；系统结合结构、证据和规则分流。

### R-04：申诉时重新调用同一 AI

- 原建议：将内容再次交给同一模型得到申诉结论。
- 决定：拒绝。
- 原因：
  - 输入不变时容易重复第一次错误；
  - 不能证明申诉纠错；
  - AI 不应直接承担终审。
- 替代：Appeal Critic 对比新旧上下文，人工最终裁决。

### R-05：两人共享其中一台电脑上的 MySQL

- 原建议：队友通过局域网长期连接一人的数据库。
- 决定：拒绝作为日常开发方式。
- 原因：
  - 主机关闭后另一人无法开发；
  - 校园网络可能隔离设备；
  - 测试数据互相污染；
  - IP 变化和防火墙增加风险。
- 替代：各自本地 MySQL + Git 迁移 + Seed；最终演示电脑统一联调。

### R-06：用多 Agent 委员会提高“AI Native”感

- 原建议：多个 Agent 讨论和投票。
- 决定：拒绝。
- 原因：
  - 多数投票不等于正确；
  - 增加延迟、成本和不确定性；
  - 固定审核流程不需要自主规划；
  - 两天内难以评测和排错。
- 替代：两个受控 AI 角色 + 确定性代码 + 人工。

## 3. 人工判断原则

团队不是按“AI 推荐次数”选择方案，而按以下顺序判断：

1. 是否解决真实问题；
2. 是否两天可交付；
3. 是否有失败兜底；
4. 是否可测试；
5. 是否可追溯；
6. 是否能明确个人贡献；
7. 是否能诚实说明边界。
