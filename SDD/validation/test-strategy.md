# 测试策略（Test Strategy）

## 1. 测试目标

验证系统不只是“页面可点”，而是能够：

- 在正常内容下完成公开；
- 在明确风险下完成限制；
- 在上下文风险下触发人工复核；
- 避免安全引用被简单误杀；
- 在 AI 失败或证据错误时安全降级；
- 允许用户申诉并由人工纠错；
- 保存完整可追溯记录；
- 修改 Prompt、规则或状态机后不破坏已通过能力。

## 2. 测试层次

| 层次 | 目标 |
| --- | --- |
| 单元测试 | Schema、证据定位、状态路由、权限、楼层号 |
| 服务测试 | Context、Moderation、Appeal、Review、Audit |
| API 测试 | 请求、响应、权限和错误码 |
| 前端交互测试 | 页面状态、角色可见性和错误提示 |
| E2E | 从发布到结果，以及从限制到申诉改判 |
| 人工评审 | 检查 AI 理由是否符合语境，而不只看状态 |

## 3. 测试类型要求

必须覆盖：

- 正常；
- 边界；
- 失败；
- AI 误判或漏判；
- 申诉复核；
- 回归；
- 至少一条 E2E；
- 至少三条题面之外的自补失败或边界样例。

## 5. 自补的隐蔽失败样例

除题面线索外，重点补充：

### 5.1 引用后赞同攻击

> “他说你是废物，我觉得没说错。”

目的：证明系统不能把所有引号都当安全语境。

### 5.2 分段连续骚扰

同一用户在不同时间发布：

```text
有人又开始表现了。
懂的都懂。
展示可别交给某些人。
```

目的：验证单句低风险、组合行为高风险。

### 5.3 AI 证据幻觉

AI 返回：

```json
{"quote": "你滚出这个群"}
```

但上下文从未出现该句。

目的：验证证据校验能阻止模型幻觉进入自动决定。

### 5.4 申诉没有新增信息

用户只写：

> “我不同意，你们审核错了。”

目的：验证系统允许申诉，但不会因为情绪化申诉自动改判。

### 5.5 楼层恢复顺序

被限制的内容申诉期间产生多个新楼层，之后申诉成功。

目的：验证恢复内容追加为当前最新楼层，而不是占用历史空号。

## 6. 端到端用例

### E2E-001：正常发布

```text
切换普通用户
→ 进入社区
→ 发起话题或发布楼层
→ AI 返回安全
→ 证据校验通过
→ 系统公开
→ 其他用户可以看到
→ 时间线存在提交、审核、公开
```

### E2E-002：限制、申诉与改判

```text
用户发布含引用的争议内容
→ 初审暂时限制
→ 内容只对作者和审核员可见
→ 用户补充引用上下文并申诉
→ Appeal Critic 输出正反依据
→ 审核员填写理由并改判允许
→ 内容成为最新公开楼层
→ 时间线保留原判与改判
```

至少 E2E-001 必须通过；高分目标是两条均通过。

## 8. 回归触发条件

以下变化必须重跑核心回归集：

- Prompt；
- 模型 Provider；
- 风险枚举；
- 上下文窗口；
- 证据校验；
- Policy Router；
- 内容状态机；
- 人工改判事务；
- 楼层号分配；
- 前后端字段。

最小回归集：

```text
TC-NORMAL-001
TC-QUOTE-001
TC-QUOTE-002
TC-HARASS-001
TC-EVIDENCE-001
TC-FAIL-001
TC-APPEAL-001
TC-REVIEW-001
TC-FLOOR-001
TC-TRACE-001
```

## 9. 测试记录要求

每次执行记录：

- 日期和执行人；
- 代码 commit；
- Prompt/规则版本；
- 环境；
- 命令；
- 预期；
- 实际；
- PASS/FAIL；
- 失败原因；
- 修复；
- 回归结果；
- 截图、日志或报告路径。

不得在未执行时提前写 PASS。
