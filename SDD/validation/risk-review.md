# 风险与反证 Review

## 1. Review 目标

主动挑战当前方案，避免团队只证明自己想证明的内容。

## 2. 关键假设反证

| 假设 | 反例 | 设计调整 |
| --- | --- | --- |
| 出现辱骂词就是违规 | 引用辱骂进行举报 | 增加引用、反驳和举报判断 |
| 没有敏感词就是安全 | 连续外号和群体施压 | 加载历史和回复对象 |
| 风险分数高就可信 | 分数高但证据不存在 | 增加证据校验 |
| 再调用一次 AI 就是申诉 | 同模型重复原错误 | Appeal Critic + 人工 |
| 审核员改判后覆盖原记录即可 | 无法解释原判和改判差异 | 分层保存和时间线 |
| 多上下文一定更好 | 整个话题带来干扰和隐私成本 | 限定相关上下文窗口 |
| 多 Agent 更准确 | 多个模型可能共享同一偏差 | 固定工作流和人工责任 |
| Mock 成功等于 AI 成功 | Mock 无法证明真实模型稳定 | 明确 Mock 边界，真实/失败均留痕 |

## 3. 最危险失败

### 3.1 高风险内容因 AI 失败被默认放行

控制：任何 Provider 异常转人工。

### 3.2 正常引用因敏感词被自动限制

控制：规则只作为信号，AI 判断引用语境，争议转人工。

### 3.3 AI 编造证据却触发限制

控制：证据必须绑定 contentId 并在原文中定位。

### 3.4 申诉系统只是重复模型调用

控制：反证角色使用新增上下文，并由人工终审。

### 3.5 过程证据与真实开发不一致

控制：

- AI 日志按发生时间填写；
- 测试未执行不得写 PASS；
- 文档计划与开发日志分开；
- Git commit 和截图作为外部佐证。

## 4. 约束变化问答

### 若时间只剩半天

保留：

- 一个固定话题；
- 发布一个楼层；
- 初审；
- 一次风险限制；
- 一次申诉改判；
- 一条时间线。

删除：

- 搜索、分类、历史列表、复杂样式和 P1。

### 若不能使用真实模型

- 使用相同 Schema 的 Mock Provider；
- 明确其只能证明工作流，不证明真实模型效果；
- 通过非法 JSON、超时和证据幻觉 Mock 验证系统兜底。

### 若要求多社区

在 P1 增加 CommunityMember 和加入权限，但不改变 Content、Moderation、Appeal 主链。

### 若要求审核后发布

增加 `published_pending_review` 和撤回机制，但需承担风险内容先扩散的代价。

### 若审核量变大

将同步调用迁移为后台任务和队列；P0 不提前引入。

## 5. Review 结论

当前方案最大的价值不是功能多，而是：

- 把 AI 放在真正需要语义判断的位置；
- 用代码约束 AI；
- 用人工承担争议责任；
- 用失败样例和审计证明系统没有假装 AI 永远正确。
