# QA Gates（质量门禁）

## 1. 目的

质量门禁用于防止：

- 只有漂亮页面，没有真实判断；
- 只有 AI 原文，没有人工选择；
- 只有结论，没有证据；
- 只演示成功路径；
- 文档、代码和测试互相矛盾；
- 成员无法解释自己的模块；
- 在答辩前继续堆无证据功能。

## 2. Gate 0：题目与范围准入

必须满足：

- [ ] 真实问题不是“敏感词过滤”；
- [ ] 用户、利益相关方、冲突、约束、假设、非目标已写明；
- [ ] ≥10 个澄清问题；
- [ ] ≥3 个 P0 澄清问题；
- [ ] 每个 P0 说明答案变化对方案的影响；
- [ ] MVP 与非目标已冻结。

不满足：禁止进入正式功能开发。

## 3. Gate 1：方案与设计准入

必须满足：

- [ ] 比较 3 个本质不同方案；
- [ ] 有选择理由和不选理由；
- [ ] 有代价、风险和兜底；
- [ ] AI、代码、人工责任边界明确；
- [ ] 服务端 + Web 前端两个技术端明确；
- [ ] 状态、数据、API、权限和失败路径有设计；
- [ ] Design 不写“已完成什么”，只写计划如何实现。

不满足：禁止拆分并行研发任务。

## 4. Gate 2：AI 协作准入

必须满足：

- [ ] 六类 AI 工作中至少五类有真实记录；
- [ ] 反证或风险 Review 有记录；
- [ ] 至少 2 条 AI 建议被拒绝或明显修改；
- [ ] 每条关键记录有目标、输入、输出摘要、采纳/拒绝、人工判断和实际影响；
- [ ] Prompt 有快照或版本；
- [ ] 不把 AI 输出直接当最终裁决。

不满足：AI 协作维度不得标记完成。

## 5. Gate 3：代码合并门禁

每个 Merge Request 必须满足：

- [ ] 对应 PRD/Design/Task；
- [ ] 修改范围清楚；
- [ ] 数据库变化有迁移；
- [ ] 前后端字段与契约一致；
- [ ] 核心逻辑有测试；
- [ ] 失败路径有处理；
- [ ] 未提交 `.env`、密钥和数据库密码；
- [ ] 另一成员完成 Review；
- [ ] Review 意见有处理记录。

不满足：不得合并到 integration/main。

## 6. Gate 4：MVP 准出

必须演示四件事：

- [ ] 一条输入到结果的主路径；
- [ ] 一次核心判断实际生效；
- [ ] 一个边界、失败或误判场景；
- [ ] 结果可回溯到输入、AI 建议和人工判断。

同时：

- [ ] 正常内容可公开；
- [ ] 风险内容不公开；
- [ ] 用户能查看自己的未公开内容；
- [ ] 用户能申诉；
- [ ] 审核员能维持或改判；
- [ ] 人工理由必填；
- [ ] 时间线保存关键节点。

不满足：不得宣称 MVP 完成。

## 7. Gate 5：验证准出

必须满足：

- [ ] 正常测试；
- [ ] 边界测试；
- [ ] 失败或误判测试；
- [ ] 回归测试；
- [ ] 至少一条 E2E；
- [ ] 至少三条自补失败或边界样例；
- [ ] AI 超时测试；
- [ ] AI 非法结构测试；
- [ ] AI 无效证据测试；
- [ ] 申诉改判测试；
- [ ] 测试结果有命令、日志或截图。

不满足：不得冻结答辩版本。

## 8. Gate 6：证据包准出

目录检查：

```text
SRC/
PROTOTYPE/
DOCS/
README.md
```

文档检查：

- [ ] diagnosis：问题、澄清、假设、非目标；
- [ ] design：端到端系统设计；
- [ ] options：三方案与取舍；
- [ ] decision：最终决策；
- [ ] validation：策略、结果、风险、Review；
- [ ] ai：AI 日志和采纳拒绝；
- [ ] collaboration：分工、会议、Issue、决策变化；
- [ ] reflection：团队复盘和个人贡献；
- [ ] PROTOTYPE：运行说明、边界、截图或录屏。

不适用内容必须写“不适用及原因”，不能保留空模板冒充完成。

## 9. Gate 7：个人贡献准出

每名成员必须能指出：

- [ ] 自己负责的关键判断；
- [ ] 自己负责的前端或后端模块；
- [ ] 自己负责的 AI Workflow 或 Prompt；
- [ ] 自己执行的测试；
- [ ] 自己提出或采纳的 Review 建议；
- [ ] 对应文件、commit、Issue 或截图；
- [ ] 对模块边界和失败路径的解释。

不满足：个人贡献进入复核。

## 10. Gate 8：答辩冻结

冻结前：

- [ ] README 可按步骤启动；
- [ ] 演示数据可复现；
- [ ] 数据库可一键迁移和 Seed；
- [ ] 主演示路径现场跑通；
- [ ] 有录屏兜底；
- [ ] 6 分钟展示脚本；
- [ ] 两人均参与回答；
- [ ] 明确当前不能证明什么；
- [ ] 准备约束变化问答；
- [ ] 不再新增大功能。

## 11. 红线

以下任一情况必须返工或进入复核：

1. 没有可运行或可交互系统；
2. 未满足服务端 + Web 前端要求；
3. 无法演示上下文审核或申诉纠错；
4. 只有最终 Demo，没有真实过程证据；
5. AI 日志或测试结果与实际明显不一致；
6. 成员无法解释自己的部分；
7. 伪造会议、AI 使用、测试或个人贡献；
8. 密钥进入仓库；
9. AI 失败时默认放行高风险内容；
10. 人工结果覆盖删除原始 AI 记录。
